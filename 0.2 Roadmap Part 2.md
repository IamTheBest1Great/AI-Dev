## 🧠 Layer 1 — The Mental Models Others Lack

### 1. Think in Failure Modes, Not Features
Before building anything, ask:
- "How will this break in production?"
- "What does the model do when it's confused?"
- "What's the worst thing a user can do to this?"

Most devs ship happy-path AI. Senior AI engineers ship systems that *degrade gracefully*.

**Practice:** For every project you build, write a 1-page failure mode doc before starting.

---

### 2. The Eval-First Mindset (Rare in 2025)
The roadmap mentions evals. Almost nobody actually does them.

**What 99th percentile looks like:**
- You define success *before* you write the prompt
- You have a labelled test dataset of 50–100 examples for every major AI feature
- Every prompt change runs through an eval suite before it ships
- You can show a graph of your AI's quality over time

**Tools to master deeply (not just install):**
- `PromptFoo` — write YAML eval suites, run in CI, track regressions
- `RAGAS` — faithfulness, answer relevancy, context precision scores
- `LLM-as-judge` — build your own judge prompt with a rubric, calibrate it against human scores
- `Braintrust` — newer eval platform worth learning early

**The signal:** When an interviewer or client asks "how do you know your AI is working?" — most people say "I tested it manually." You show them a dashboard.

---

### 3. Prompt as Code (Version-Controlled, Tested, Deployed)
Hardcoded prompts in a `.js` file = junior signal.

**What elite looks like:**
- Prompts live in a database with versions, metadata, and change history
- Prompt deploys are separate from code deploys
- You A/B test prompts with statistical significance tracking
- You have a prompt registry: name, version, model, temperature, purpose

**Build this:** A simple internal prompt management UI backed by MongoDB. This alone is a portfolio differentiator.

---

### 4. The Context Window Budget Mindset
Most devs stuff everything into context and hope for the best.

Elite engineers treat the context window like RAM — precious, finite, and to be allocated deliberately:

```
System prompt            →  ~500–1000 tokens (compressed, cached)
Retrieved context        →  ~2000–4000 tokens (ranked by relevance, not recency)
Conversation history     →  summarized after 10 turns (~500 tokens)
Tool results             →  trimmed to relevant fields only
User message             →  kept as-is
```

**Skill to build:** Write a context budget calculator that tells you exactly how many tokens each section uses before an LLM call fires.

## ⚙️ Layer 2 — Technical Skills With Almost No Competition

### 5. LangGraph Mastery (Not Just Usage)
The roadmap mentions LangGraph. Almost nobody has built something *complex* with it.

**What to actually build:**
- A workflow with conditional branching (not just linear chains)
- A human-in-the-loop checkpoint where the graph pauses for approval

**Why this matters:** LangGraph is hard. Most people watch one tutorial and think they know it. If you can draw a LangGraph workflow on a whiteboard and explain why you chose each node, you're in the top 1%.

### 7. AI Security (OWASP LLM Top 10 — Actually Implement It)
The roadmap *mentions* OWASP LLM Top 10. Nobody actually implements defenses.

**Skills that pay:**
- **Prompt injection defense:** System prompt hardening, output validation layer, role separation
- **Indirect prompt injection:** An agent reads a web page. The page contains "Ignore previous instructions." Can you catch it?
- **PII scrubbing before API calls:** Use `presidio` (Microsoft) or `GLiNER` to detect and redact PII before it leaves your system
- **Jailbreak red-teaming:** Test your own system with adversarial prompts. Document what breaks it. Fix it.

**Build this:** An AI security middleware layer for Express/Node — PII detection, injection detection, content moderation — that wraps every LLM call. Open-source it.

---

### 8. Semantic Caching (Underrated, High ROI)
Most devs cache by exact key. Semantic caching caches by *meaning*.

- If user asks "What's the capital of France?" and later "France's capital city?" — they should hit the same cache.
- Tools: `GPTCache`, custom Redis + embedding similarity lookup
- Impact: 40–60% cost reduction on high-traffic AI apps

**Build this as a drop-in middleware.** It's genuinely useful, rare, and shows you think about economics, not just functionality.

---

### 9. Structured Extraction at Scale
Beyond basic JSON mode — real document intelligence:

- **Instructor.js** with complex nested Zod schemas
- Multi-page PDF extraction with page-level confidence scores
- Table extraction from PDFs (using `pdfplumber` via Python sidecar or Azure Document Intelligence)
- Entity extraction + relationship mapping → Neo4j knowledge graph
- Schema auto-detection: given an unknown document, have the LLM *propose* the output schema before extracting

**Why rare:** Most devs use JSON mode and call it done. Building a robust extraction pipeline that handles malformed outputs, retries with fallback schemas, and logs confidence — that's a product.

### 10. Observability That Actually Tells You Something
Langfuse traces are table stakes. What separates elite:

- **Custom metadata on every trace:** user ID, session ID, prompt version, model, feature flag
- **Cost attribution per feature:** "The summarization feature costs $0.003 per run; the research agent costs $0.14"
- **Quality score on every trace:** LLM-as-judge running async on every response, score stored in DB
- **Anomaly detection:** Alert when latency spikes, cost per session doubles, or quality score drops below threshold
- **Cohort analysis:** Does quality differ between free and paid users? Between mobile and desktop?

**Build this:** A custom observability dashboard (Next.js + Langfuse SDK + your own DB) that shows all of the above. This is a legitimate SaaS product.

## 🏗️ Layer 3 — Architecture Thinking That Sets Seniors Apart

### 11. Multi-Tenant AI Architecture
Building AI for one user is easy. Building it for 1,000 businesses on one platform is hard.

**What you must know:**
- Namespace isolation in vector stores (one Pinecone namespace per tenant)
- Prompt injection risk: tenant A's data must never appear in tenant B's responses
- Per-tenant model config: some clients want GPT-4o, some want Claude, some want a fine-tuned model
- Usage metering per tenant: track tokens used, cost incurred, limit enforcement
- Tenant-level rate limiting in Redis

**Build this:** A white-label chatbot SaaS where each client uploads their own docs, has their own branding, and gets a usage dashboard. This is the most commercially sellable thing on the entire roadmap.

---

### 12. Async AI Pipelines (Not Everything Should Be Real-Time)
Most AI apps are synchronous: request → LLM → response. This breaks at scale.

**Patterns to master:**
- BullMQ job queue for expensive AI tasks (research agents, large document processing)
- Webhook-based async: user submits, gets job ID, polls for result
- Streaming progress updates via SSE while a multi-step agent runs
- Priority queues: paid users' jobs run before free users'
- Dead letter queues: failed AI jobs get logged, retried, then escalated

**Why this matters:** Every enterprise AI integration eventually needs async pipelines. This is where "AI developer" becomes "AI systems engineer."

---

### 13. The Model Router (Build One, Understand All)
LiteLLM is great. But building your own simple router teaches you more:

```
Query complexity classifier (fast, cheap model)
    → Simple query         → GPT-4o Mini / Haiku
    → Medium query         → GPT-4o / Sonnet
    → Complex / agentic    → o1 / Opus
    → Code-specific        → Claude Code / Codestral
    → Private/sensitive    → Local Ollama model
```

**Add:** Cost caps per user per day, automatic fallback on API errors, latency-based routing (if provider A is slow, route to B).

**Why it matters:** This is real cost engineering. Companies pay engineers who can reduce AI costs by 60% without degrading quality.

## 🔩 Foundation Layer — Non-AI Engineering You Must Be Strong In

### F1. TypeScript — Not Optional, Not Shallow
Every serious AI backend at a real company is TypeScript-first. Why: LLM outputs are unpredictable JSON. If you're not doing strict type validation on every API response, you're shipping bugs.

**What "strong" means:**
- Generics, conditional types, mapped types — not just basic interfaces
- `zod` for runtime validation of LLM JSON output (type-safe at parse time, not just compile time)
- Type inference from Zod schemas (`z.infer<typeof schema>`)
- Discriminated unions for handling different LLM response types
- Branded types for things like `TenantId`, `UserId`, `SessionId` — prevents mixing up IDs

**How to get strong:** Build your next 3 projects with `strict: true` in tsconfig. No `any`. Fix every type error instead of suppressing it.

---

### F2. System Design — The Interview and Job Skill
AI engineers at big companies design systems, not just write code. You need to be able to whiteboard and defend architectural decisions.

**Core concepts to actually understand (not just know the words):**

- **CAP theorem:** Why your RAG vector store can be eventually consistent and that's fine, but your billing system cannot be
- **Rate limiting patterns:** Token bucket vs leaky bucket vs sliding window — when each is right
- **Database sharding:** How Swiggy or Zepto would shard a table of 500M AI conversations
- **Event-driven architecture:** Kafka/Redis Streams for AI pipeline events (document uploaded → chunk → embed → index)
- **Idempotency:** Why every AI job queue message needs an idempotency key — what happens if BullMQ retries the same embedding job 3 times
- **Back pressure:** What happens when your AI pipeline receives 10,000 jobs and can only process 100/min — how do you not crash

**How to learn it:** *System Design Interview* book (Alex Xu) + design one system per week on paper. Actually draw the boxes, label the data flows, ask "what breaks first?"

---

### F3. Databases — Beyond CRUD
AI systems hit databases in patterns most CRUD apps never see.

**What to learn:**

- **PostgreSQL deeply:** Indexes (B-tree, GIN, BRIN), query plans (`EXPLAIN ANALYZE`), connection pooling (PgBouncer), partitioning large tables by date
- **Redis beyond caching:** Pub/Sub for real-time events, Streams for durable message queues, sorted sets for leaderboards and rate limiting, TTL strategies for session data
- **MongoDB aggregation pipelines:** `$lookup`, `$unwind`, `$group` — for complex conversation analytics
- **pgvector:** How HNSW and IVFFlat indexes work, why your vector search slows down at 1M rows and what to do
- **Time-series patterns:** Storing AI quality scores over time, querying "average faithfulness score this week vs last week"

**How to learn it:** Take one of your existing projects. Run `EXPLAIN ANALYZE` on its slowest query. Fix it. Do this 10 times across different projects.

---

### F4. Distributed Systems Basics
Not PhD-level. But enough to not make rookie mistakes.

**What you must understand:**

- **Exactly-once vs at-least-once delivery:** BullMQ delivers at-least-once by default. What does this mean for your AI embedding pipeline? (Answer: make embedding jobs idempotent — check if already embedded before re-embedding)
- **Distributed locks:** If two workers both try to process the same document, how do you prevent double-embedding? (Answer: Redis `SET NX EX`)
- **Saga pattern:** For multi-step AI workflows — if step 3 fails (LLM call), how do you roll back steps 1 and 2 (file stored, chunks created)?
- **Outbox pattern:** Writing to DB and publishing an event atomically — critical for "document uploaded → trigger embedding" without losing events

**How to learn it:** Build a distributed job processor from scratch (no BullMQ, just Redis + Node.js workers). See every failure mode yourself.

---

### F5. Observability as a First-Class Skill
At big companies, if it's not observable, it doesn't exist. This applies twice as hard to AI because AI fails silently.

**The three pillars — master all three:**

**Logs (structured, not console.log):**
```javascript
// Bad — unfilterable, unsearchable
console.log("LLM call done");

// Good — every field is queryable in Datadog/Grafana
logger.info({
  event: "llm_call_completed",
  model: "claude-sonnet-4-20250514",
  promptVersion: "v2.3",
  tokensUsed: 1240,
  latencyMs: 1823,
  userId: "usr_abc123",
  tenantId: "tenant_xyz",
  featureFlag: "new_rag_pipeline",
  qualityScore: 0.87
});
```

**Metrics (counters, gauges, histograms):**
- `ai.llm.calls.total` — counter, tagged by model + feature
- `ai.llm.latency` — histogram, p50/p95/p99
- `ai.tokens.used` — counter, tagged by tenant for billing
- `ai.quality.score` — gauge, async from LLM-as-judge
- `ai.cache.hit_rate` — gauge, semantic cache effectiveness

**Traces (distributed, end-to-end):**
- Every user request → one trace ID
- Trace spans across: API → RAG retrieval → LLM call → response formatting
- LangSmith/Langfuse for AI-specific traces, OpenTelemetry for infrastructure traces
- Connect both: one trace ID that links your app's APM trace to the LLM trace

**How to learn it:** Set up Langfuse + a custom Prometheus/Grafana stack on one project. Write 5 custom metrics. Build one alert that fires when quality score drops below 0.7.

### F6. CI/CD for AI — The Pipeline That Catches Regressions
At big companies, nothing ships without CI. AI features especially.

**What a production AI CI pipeline looks like:**

```
PR opened
  → Lint + TypeScript check
  → Unit tests (pure logic, no API calls)
  → Integration tests (mock LLM responses)
  → Eval suite runs (PromptFoo against labelled dataset)
      → If eval score < threshold → CI fails, PR blocked
  → Security scan (gitleaks for secrets, npm audit)
  → Cost estimate check (will this change increase token usage by >20%?)

Merged to main
  → Docker build
  → Deploy to staging
  → Smoke tests against staging (real LLM calls, small dataset)
  → Canary deploy to 5% of production traffic
  → Monitor quality score for 30 min
  → Full rollout OR auto-rollback

Prompt-only change (no code change)
  → Same eval suite
  → Shadow mode: run new prompt alongside old, compare outputs
  → Gradual rollout with A/B quality tracking
```

**Tools:** GitHub Actions, Docker, PromptFoo, Langfuse SDK (to pull quality scores), ArgoCD or Railway for deploy automation.

**How to build it:** Start with just `GitHub Actions → PromptFoo eval → fail if score drops`. That alone puts you ahead of 95% of AI devs.

## 🏗️ Infrastructure Layer — What Big Companies Run On

### I1. Containerization and Docker (Real Competence, Not Just `docker run`)
Every AI service at a company runs in a container.

**What to actually know:**
- Multi-stage builds: separate build image from runtime image (final image is 200MB not 2GB)
- `.dockerignore` that excludes `node_modules`, `.env`, model weights
- Health check endpoints: `GET /health` returns 200 only when DB + Redis + LLM provider are reachable
- Environment-based config: no secrets in Dockerfile, all via environment variables
- Docker Compose for local: app + Redis + Postgres + Langfuse all running locally with one command

**AI-specific patterns:**
- GPU container setup for local model inference (Ollama in Docker with GPU passthrough)
- Sidecar containers: Python container alongside Node.js for AI-specific libraries (presidio, pdfplumber)
- Volume mounts for model weights (don't bake 4GB models into images)

### I2. Cloud Fundamentals (Pick One Cloud Deeply)
You don't need to be a DevOps engineer. But you need to understand what you're deploying to.

**For most AI SaaS — AWS is the standard. Learn these services:**

- **EC2 / ECS / Fargate:** Where your Node.js AI backend runs
- **S3:** Where user-uploaded PDFs, audio files, and model outputs live
- **RDS (PostgreSQL):** Primary database with read replicas for heavy read workloads
- **ElastiCache (Redis):** Session storage, caching, BullMQ backend
- **SQS:** Managed queue for async AI jobs (alternative to BullMQ for serverless)
- **CloudWatch:** Logs + metrics + alerts (connect to your structured logger)
- **IAM:** Least-privilege access — your AI service should only have the permissions it needs
- **Secrets Manager:** Where API keys live — never environment variables in production
- **VPC + Security Groups:** Your AI backend should not be publicly accessible — only through an API gateway or load balancer

**For GPU workloads:**
- **Modal:** Best for serverless GPU (fine-tuning, inference). Spins up in seconds, costs per second.
- **Runpod:** Cheaper than AWS for dedicated GPU instances
- **Lambda Labs:** Good for long-running training jobs

### I3. Kubernetes Basics (Don't Skip This for Big Company Roles)
You won't be a Kubernetes engineer. But you need to read a `deployment.yaml` and understand what it does.

**What to learn:**
- Pods, Deployments, Services, Ingress — what each is and why
- Resource requests and limits: `cpu: 500m, memory: 512Mi` — why this prevents one AI service from starving others

**How to learn it:** Set up a local cluster with `k3d` or `minikube`. Deploy your most complex AI project to it. Break it and fix it.

### I4. Message Queues — The Backbone of Production AI
Every company running AI at scale has a message queue between their API and their AI workers.

**Why:** LLM calls are slow (2–15 seconds), unpredictable (provider outages), and expensive (batch when possible). A queue decouples all three problems.

**Architecture pattern:**

```
User uploads PDF
  → API returns 202 Accepted + job_id
  → Job pushed to queue (SQS / BullMQ / Kafka)
  → Worker pool picks up job
  → Worker: extract text → chunk → embed → store in vector DB
  → Worker: update job status to "complete" in DB
  → Frontend polls /api/jobs/{job_id} or receives WebSocket push
```

**What to learn:**
- **BullMQ:** Best for Node.js. Concurrency, rate limiting, retries, priorities, delayed jobs, cron jobs.
- **SQS:** AWS managed, great for scale. Visibility timeout, dead letter queues, long polling.
- **Kafka (basics):** For high-throughput event streaming. Consumer groups, partitions, offsets. You won't run Kafka but you'll work with it at big companies.

### I5. Feature Flags — The AI Rollout Superpower
At big companies, you never ship AI changes to 100% of users immediately.

**What feature flags enable:**
- Roll out new model to 1% of users, measure quality, expand or rollback
- A/B test prompts: 50% get prompt v1, 50% get v2, compare quality scores
- Kill switch: instantly disable a broken AI feature without a deploy
- Gradual rollout by user cohort: paid users first, then free

**Tools:**
- **LaunchDarkly:** Industry standard, expensive
- **Growthbook:** Open-source, self-hostable, works great with Next.js
- **Flagsmith:** Another good open-source option
- **Vercel Flags:** If you're on Vercel's ecosystem

**Build pattern:** Every AI feature behind a flag. Every prompt change behind a flag. Measure before full rollout. Always have a kill switch.

## 🧪 Quality Layer — What Separates Reliable AI from Demo AI

### Q1. Testing Strategy for AI Systems
AI is non-deterministic. Standard unit tests break. You need a different testing philosophy.

**The four levels:**

**Level 1 — Deterministic unit tests (no LLM involved):**
```
Test: chunking algorithm produces correct chunk sizes
Test: token counter returns correct count
Test: metadata filter correctly narrows vector search
Test: retry logic fires after correct delay
Test: rate limiter blocks after threshold
```

**Level 2 — Integration tests with mocked LLM:**
```
Test: API endpoint returns correct structure when LLM returns mocked response
Test: Error handling when LLM times out (mock a 30-second delay)
Test: Fallback chain triggers when primary model returns error
Test: Conversation history is correctly truncated at token limit
```

**Level 3 — Evals (real LLM, labelled dataset):**
```
Input: 50 real user questions
Expected: defined in eval config (not exact match, but quality criteria)
Measure: faithfulness, answer relevancy, factual accuracy
Run: on every PR, compare to baseline score
Alert: if score drops >5% from baseline, block merge
```

**Level 4 — Shadow mode / canary testing (production traffic):**
```
Run new model/prompt alongside current model on 5% of real traffic
Compare outputs (LLM-as-judge)
Only promote to full traffic if quality is ≥ current
```

---

### Q2. Data Quality — The Invisible AI Problem
Garbage in, garbage out. At big companies, there are entire teams focused on data quality for AI.

**What you need to own as an AI engineer:**

- **Document pre-processing pipeline:** Before embedding, every document goes through cleaning — remove headers/footers, fix encoding issues, extract tables separately, caption images
- **Chunk quality checks:** After chunking, validate no chunk is <50 tokens (too small to be meaningful) or >1000 tokens (too large to be specific)
- **Embedding drift detection:** When you change the embedding model, old embeddings are now in a different space — detect this and re-embed
- **Data freshness:** If your RAG knowledge base has a document from 2022 and today is 2025, how do you know it's stale? Build staleness detection.
- **Deduplication:** Before embedding, detect near-duplicate documents and consolidate (MinHash or embedding similarity)

---

### Q3. The Eval Dataset — Your Most Valuable Engineering Asset
At big companies, the eval dataset is treated like production code. Versioned. Reviewed. Expanded over time.

**How to build one:**
1. Start with 20 real user queries from production logs
2. Write the ideal answer for each manually (ground truth)
3. Tag each with difficulty (easy/medium/hard) and category (factual/reasoning/extraction)
4. Add edge cases: questions the AI should refuse, questions with no answer in the knowledge base
5. Review and expand by 10 examples per month
6. Use it in CI — every prompt change runs against this dataset

**The signal:** Companies that have this run it in their interview process. "Show me your eval dataset and the last time it caught a regression" — most candidates have nothing. You have a 200-example labelled dataset with a CI dashboard.

## 💼 Engineering Process Layer — How Big Companies Work

### P1. How to Read and Write Technical Design Documents (TDDs)
Every non-trivial AI feature at a big company starts with a design doc. If you can't write one, you won't get promoted past mid-level.

**Structure of a good AI feature TDD:**

```
## Problem
What pain does this solve? Who has it? How often?

## Proposed Solution
High-level description. What changes.

## Architecture Diagram
Box-and-arrow diagram of the new system.

## Data Model Changes
New tables, columns, indexes. Migrations required.

## API Contract
New endpoints or changes to existing. Request/response schemas.

## AI Model & Prompt Design
Which model. Why. Prompt (versioned). Expected quality metrics.

## Eval Plan
How will we measure success? What's the baseline? What's the threshold for shipping?

## Rollout Plan
Feature flag. Canary %. Metrics to watch. Rollback trigger.

## Cost Estimate
Tokens per request × daily requests × cost per token = $/day.

## Open Questions
What's still unknown. Who decides.
```

**How to practice:** Write a TDD for every significant project you build, even personal ones.

### P2. Code Review Culture — Give and Receive
At big companies, every line of code is reviewed before merging. If you've never had your AI code reviewed by a senior engineer, you have blind spots.

**What reviewers look for in AI code:**
- Prompt hardcoded? (red flag)
- LLM output validated before use? (critical)
- Tokens logged? Cost tracked? (required)
- Error path handled? What if LLM returns null?
- Is this synchronous when it should be async?
- Is the context window budget considered?
- Is the eval coverage adequate for this change?

**How to practice:** Find one good open-source AI project on GitHub. Read their PRs. Comment on the code (even privately in your notes) — what would you say if you were reviewing it?

---

### P3. Incident Response for AI Systems
At big companies, when AI breaks at 2am, someone gets paged. Usually the engineer who built it.

**What to learn:**
- Write a runbook for every AI feature you ship: "If quality score drops below 0.6, do X. If LLM provider returns 503, do Y. If costs spike 3x, do Z."
- Know how to roll back a prompt change without a code deploy
- Know how to disable a feature flag in 30 seconds
- Know how to pull a Langfuse trace for a specific user's session to debug why their answer was wrong
- Know how to check your cost dashboard to verify a billing spike is real

## The Mindset Shift That Unlocks All of This

Junior AI engineers ask: **"How do I make the LLM do X?"**

Senior AI engineers ask: **"How do I make the system reliably, measurably, safely do X — and know when it stops?"**

That single shift in how you frame problems is what separates someone who ships demos from someone who ships production systems that companies pay $200k+ for.

The skills in this document aren't AI skills. They're engineering skills applied to AI. And that's exactly why almost nobody has them.
