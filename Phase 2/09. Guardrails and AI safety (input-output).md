# 📘 Guardrails & AI Safety (Input/Output Validation) — Complete Guide

As you build AI‑powered applications, ensuring **safety, reliability, and trust** becomes critical. Guardrails are the policies, checks, and filters you put in place to control what goes into your LLM and what comes out of it. This guide covers the fundamentals of AI safety, common threats, and practical implementation techniques for input/output validation.

---

## 1. Why Guardrails Matter

LLMs are powerful but unpredictable. Without guardrails, you risk:

- **Prompt injection** – Malicious users trick the model into ignoring instructions or revealing secrets.
- **Toxic content** – The model may generate hate speech, violence, or inappropriate material.
- **PII leakage** – The model could inadvertently output personal identifiable information (emails, phone numbers, etc.).
- **Hallucinations** – The model invents facts, which can mislead users.
- **Brand damage** – Unfiltered output may contain offensive or unprofessional language.
- **Legal liability** – Violations of regulations (GDPR, CCPA, etc.) if PII is exposed.

Guardrails act as a safety layer between the user, your application, and the LLM.

---

## 2. Types of Guardrails

| Type | Description | Examples |
|------|-------------|----------|
| **Input validation** | Checks on user input before it reaches the LLM | Blocking jailbreak attempts, redacting PII, limiting topic scope |
| **Output validation** | Checks on LLM responses before showing to user | Toxicity filters, fact‑checking, PII redaction |
| **Moderation** | Pre‑defined category checks (violence, hate, self‑harm) | OpenAI Moderation API, custom classifiers |
| **Structural validation** | Enforcing output format (JSON, markdown, etc.) | Zod schemas, regex patterns |
| **Semantic validation** | Ensuring output makes sense or stays on topic | Cosine similarity with expected topics, chain‑of‑thought verification |

## 3. Guardrails Implementation Approaches

### 3.1 Simple Custom Validation (Code‑Based)

You can implement lightweight guardrails directly in your application logic.

#### Input Validation Examples

```javascript
// Block obvious prompt injection keywords
const blockedPatterns = [
  /ignore previous instructions/i,
  /forget all rules/i,
  /you are now/i,
  /system prompt/i
];

function validateInput(userInput) {
  for (const pattern of blockedPatterns) {
    if (pattern.test(userInput)) {
      throw new Error('Input contains disallowed patterns');
    }
  }
  return true;
}
```

#### Output Validation Examples

```javascript
// Redact emails and phone numbers from output
function redactPII(text) {
  const emailRegex = /[\w.-]+@[\w.-]+\.\w+/g;
  const phoneRegex = /\b\d{3}[-.]?\d{3}[-.]?\d{4}\b/g;
  return text
    .replace(emailRegex, '[EMAIL REDACTED]')
    .replace(phoneRegex, '[PHONE REDACTED]');
}

// Check for toxicity using a simple word blocklist
const toxicWords = ['hate', 'kill', 'stupid']; // extend as needed
function containsToxic(text) {
  return toxicWords.some(word => text.toLowerCase().includes(word));
}
```

### 3.3 OpenAI Moderation API

OpenAI provides a free moderation endpoint that classifies content into categories: hate, harassment, violence, self‑harm, sexual, etc.

```javascript
import OpenAI from 'openai';

const openai = new OpenAI();

async function moderate(content) {
  const response = await openai.moderations.create({ input: content });
  const result = response.results[0];
  if (result.flagged) {
    console.log('Flagged categories:', result.categories);
    return false; // block
  }
  return true;
}

// Usage before showing output to user
const isSafe = await moderate(llmOutput);
if (!isSafe) {
  // show fallback message
}
```

### 3.4 Structured Outputs with Zod

Forcing the LLM to return a well‑defined JSON structure reduces hallucinations and makes validation easy.

```javascript
import { z } from 'zod';
import OpenAI from 'openai';

const openai = new OpenAI();

const RecipeSchema = z.object({
  name: z.string(),
  ingredients: z.array(z.string()),
  steps: z.array(z.string()),
  prepTimeMinutes: z.number().optional()
});

async function extractRecipe(userPrompt) {
  const completion = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      { role: 'system', content: 'Return a JSON matching the schema.' },
      { role: 'user', content: userPrompt }
    ],
    response_format: { type: 'json_object' }
  });

  const raw = completion.choices[0].message.content;
  try {
    const parsed = JSON.parse(raw);
    return RecipeSchema.parse(parsed); // throws if invalid
  } catch (e) {
    console.error('Output validation failed:', e);
    return null; // fallback
  }
}
```

### 3.6 Chaining Multiple Guards

A robust pipeline may combine several checks:

```javascript
async function safeLLMResponse(userInput) {
  // 1. Input validation
  if (!validateInput(userInput)) {
    return { error: 'Invalid input' };
  }

  // 2. Optional: pre‑moderation of user input
  const inputSafe = await moderate(userInput);
  if (!inputSafe) {
    return { error: 'Input flagged' };
  }

  // 3. Call LLM (with safe prompt)
  const llmOutput = await callLLM(userInput);

  // 4. Post‑process: redact PII
  const redacted = redactPII(llmOutput);

  // 5. Check output moderation
  const outputSafe = await moderate(redacted);
  if (!outputSafe) {
    return { error: 'Output flagged' };
  }

  // 6. Return safe output
  return { success: true, data: redacted };
}
```

## 4. Common Threat Vectors and Mitigations

| Threat | Description | Mitigation |
|--------|-------------|------------|
| **Prompt injection** | User tries to override system prompt | Use strong system prompt; filter input for injection patterns; use a separate LLM to detect attacks. |
| **Jailbreaking** | Adversarial prompts designed to bypass safety | Multi‑layer validation; use moderation APIs; limit context window. |
| **PII leakage** | Model reveals private data | Train model to not output PII; use post‑processing redaction (Presidio). |
| **Toxic content** | Hate, violence, harassment | OpenAI Moderation API; custom blocklists; human review for critical apps. |
| **Hallucinations** | Model invents false information | RAG (ground in data); ask model to cite sources; use fact‑checking tools. |
| **Off‑topic** | Model strays from allowed topics | Classify intent before generation; use guardrails to restrict topic. |

---

## 5. Best Practices

- **Defense in depth** – Combine multiple layers: input validation, prompt design, output filtering.
- **Least privilege** – Give the model only the tools and context it absolutely needs.
- **Monitor and log** – Track violations to improve your guardrails over time.
- **Fallback responses** – When something is blocked, provide a helpful default message.
- **Human in the loop** – For high‑stakes decisions, have a human review flagged content.
- **Regular updates** – Threats evolve; update blocklists and models regularly.

## 8. Example: Full Guardrail Wrapper in TypeScript

```typescript
import OpenAI from 'openai';
import { z } from 'zod';
import { createHash } from 'crypto';

const openai = new OpenAI();

// Simple in‑memory cache to avoid re‑moderating identical content
const moderationCache = new Map<string, boolean>();

async function moderateWithCache(content: string): Promise<boolean> {
  const hash = createHash('sha256').update(content).digest('hex');
  if (moderationCache.has(hash)) return moderationCache.get(hash)!;
  const response = await openai.moderations.create({ input: content });
  const flagged = response.results[0].flagged;
  moderationCache.set(hash, !flagged);
  return !flagged;
}

function redactPII(text: string): string {
  const emailRegex = /[\w.-]+@[\w.-]+\.\w+/g;
  const phoneRegex = /\b\d{3}[-.]?\d{3}[-.]?\d{4}\b/g;
  return text.replace(emailRegex, '[EMAIL]').replace(phoneRegex, '[PHONE]');
}

const OutputSchema = z.object({
  summary: z.string(),
  keyPoints: z.array(z.string()),
  sentiment: z.enum(['positive', 'neutral', 'negative'])
});

export async function safeAnalyze(text: string) {
  // 1. Input moderation
  if (!(await moderateWithCache(text))) {
    throw new Error('Input contains inappropriate content.');
  }

  // 2. Call LLM with structured output
  const completion = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      { role: 'system', content: 'You are a helpful analyst. Return valid JSON.' },
      { role: 'user', content: text }
    ],
    response_format: { type: 'json_object' }
  });

  const raw = completion.choices[0].message.content;
  if (!raw) throw new Error('No output');

  // 3. Parse and validate structure
  let parsed;
  try {
    parsed = OutputSchema.parse(JSON.parse(raw));
  } catch {
    throw new Error('Output format invalid');
  }

  // 4. Redact PII from all text fields
  const redactedSummary = redactPII(parsed.summary);
  const redactedPoints = parsed.keyPoints.map(redactPII);

  // 5. Output moderation
  const outputText = redactedSummary + ' ' + redactedPoints.join(' ');
  if (!(await moderateWithCache(outputText))) {
    throw new Error('Output flagged by moderation.');
  }

  return {
    summary: redactedSummary,
    keyPoints: redactedPoints,
    sentiment: parsed.sentiment
  };
}
```

## 9. Summary

Guardrails are not optional – they are a fundamental part of building trustworthy AI applications. By implementing input/output validation, moderation, and structural checks, you protect your users, your brand, and your legal standing.

**Key takeaways**:
- Start with simple custom checks and evolve.
- Use specialized tools (Moderation API, Presidio, Zod) for robust protection.
- Always validate both input and output.
- Design fallbacks for when content is blocked.
- Keep learning – the threat landscape changes quickly.
