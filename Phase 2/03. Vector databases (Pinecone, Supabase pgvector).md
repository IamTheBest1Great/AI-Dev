# 📘 Vector Databases (Pinecone, Supabase pgvector) — Complete Guide

This guide covers the third topic in Phase 2: Vector Databases. You've learned what embeddings are and how to generate them via API—now it's time to store and query them efficiently.

---

## 1. What Is a Vector Database?

A **vector database** is a specialized storage system designed to efficiently store, index, and query high-dimensional vectors (embeddings) based on **similarity** rather than exact matching.

### Core Features
- **Approximate Nearest Neighbor (ANN) search** – Finds similar vectors without exhaustive comparison
- **Indexing algorithms** – HNSW, IVF, LSH for fast retrieval
- **Metadata filtering** – Combine semantic search with structured filters
- **CRUD operations** – Manage vectors and associated metadata

### Why Not Just Use a Regular Database?
Traditional databases excel at exact matches (`WHERE age > 25`), but vector search requires finding "the 10 most similar items" – a fundamentally different operation that needs specialized indexing.

---

## 2. How Vector Databases Work

### The Core Challenge
When you have millions of vectors, computing exact distances for every query is impossible. Vector databases use **ANN algorithms** that trade perfect accuracy for speed:
- **95%+ recall** while being orders of magnitude faster
- **Milliseconds response time** even with billions of vectors

### Popular Indexing Algorithms

| Algorithm | Description | Best For |
|-----------|-------------|----------|
| **HNSW** (Hierarchical Navigable Small World) | Creates multi-layer graph structure | Best accuracy-speed balance |
| **IVF** (Inverted File Index) | Partitions vectors into clusters | Large-scale datasets |
| **LSH** (Locality-Sensitive Hashing) | Hashes similar items into same buckets | Fastest, less precise |

## 3. Pinecone: Fully Managed Vector Database

Pinecone is a leading **fully managed** vector database that handles infrastructure, scaling, and maintenance automatically.

### Key Advantages
- **Zero运维** – No infrastructure to manage
- **Real-time updates** – New vectors indexed instantly
- **Scale to billions** – From zero to billions with minimal latency impact
- **High performance** – Low query latency even at massive scale

### Getting Started with Pinecone

#### Step 1: Create Account and Get API Key
1. Sign up at [pinecone.io](https://www.pinecone.io/)
2. Get API key from dashboard
3. Install SDK: `npm install pinecone` (Node.js) or `pip install pinecone` (Python)

#### Step 2: Create an Index

**Python Example** (with integrated embeddings):
```python
from pinecone import Pinecone

pc = Pinecone(api_key="your-api-key")

# Create index with integrated embedding model
index_name = "my-docs"
if not pc.has_index(index_name):
    pc.create_index_for_model(
        name=index_name,
        cloud="aws",
        region="us-east-1",
        embed={
            "model": "llama-text-embed-v2",
            "field_map": {"text": "chunk_text"}
        }
    )
```

#### Step 3: Upload Documents
```python
dense_index = pc.Index(index_name)

records = [
    {
        "_id": "rec1",
        "chunk_text": "The Eiffel Tower was completed in 1889...",
        "category": "history"
    },
    {
        "_id": "rec2", 
        "chunk_text": "Photosynthesis converts light to chemical energy...",
        "category": "science"
    }
]

dense_index.upsert_records("my-namespace", records)
```

#### Step 4: Semantic Search
```python
query = "Famous historical monuments"
results = dense_index.search(
    namespace="my-namespace",
    query={
        "top_k": 10,
        "inputs": {'text': query}
    }
)

for hit in results['result']['hits']:
    print(f"Score: {hit['_score']:.2f}, Text: {hit['fields']['chunk_text'][:50]}...")
```

#### Step 5: Advanced Features

**Metadata Filtering**:
```python
results = dense_index.search(
    namespace="my-namespace",
    query={"top_k": 10, "inputs": {'text': query}},
    filter={"category": {"$eq": "history"}}
)
```

**Result Reranking** (improves relevance):
```python
reranked = dense_index.search(
    namespace="my-namespace",
    query={"top_k": 10, "inputs": {'text': query}},
    rerank={
        "model": "bge-reranker-v2-m3",
        "top_n": 10,
        "rank_fields": ["chunk_text"]
    }
)
```

**Multi-tenancy with Namespaces**:
```python
# Isolate data for different users/customers
dense_index.upsert_records("user-123", user_records)
dense_index.upsert_records("user-456", other_user_records)
```

## 4. Supabase pgvector: Vector Search in PostgreSQL

**pgvector** is a PostgreSQL extension that adds vector similarity search capabilities to your existing database. Unlike Pinecone's managed service, pgvector runs inside your Supabase (or any PostgreSQL) database.

### Key Advantages
- **Part of your existing database** – No new infrastructure
- **ACID compliance** – Transactions, foreign keys, backups
- **SQL integration** – Combine vector search with relational queries
- **Cost-effective** – Use your existing PostgreSQL instance

### Getting Started with pgvector in Supabase

#### Enable the Extension
```sql
-- In Supabase dashboard: Database → Extensions → enable "vector"
-- Or via SQL:
create extension if not exists vector;
```

#### Create a Table with Vector Column
```sql
create table posts (
    id serial primary key,
    title text not null,
    body text not null,
    embedding vector(384)  -- 384-dimensional embeddings
);
```

#### Generate and Store Embeddings (Node.js)
```javascript
import { pipeline } from '@xenova/transformers';
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(process.env.SUPABASE_URL, process.env.SUPABASE_KEY);
const generateEmbedding = await pipeline('feature-extraction', 'Supabase/gte-small');

async function storePost(title, body) {
    // Generate embedding
    const output = await generateEmbedding(body, {
        pooling: 'mean',
        normalize: true,
    });
    const embedding = Array.from(output.data);
    
    // Store in database
    const { data, error } = await supabase
        .from('posts')
        .insert({ title, body, embedding });
    
    return data;
}
```

#### Perform Similarity Search
```sql
-- Find posts similar to a query vector
select 
    id,
    title,
    body,
    1 - (embedding <=> '[0.12, -0.34, ...]') as similarity
from posts
order by embedding <=> '[0.12, -0.34, ...]'
limit 10;
```

**Distance operators**:
- `<=>` – Cosine distance (most common for text)
- `<->` – Euclidean (L2) distance
- `<#>` – Negative inner product

#### Combining with Metadata Filters
```sql
-- Find similar posts in a specific category
select *
from posts
where category = 'technology'
order by embedding <=> '[0.12, -0.34, ...]'
limit 10;
```

## 5. Automatic Embeddings with Supabase (Production Setup)

Supabase provides a complete architecture for **automatically generating embeddings** when content changes, using Edge Functions, queues, and cron jobs.

### Architecture Components
- **pgvector** – Stores embeddings
- **pgmq** – Queues embedding jobs
- **pg_net** – Async HTTP requests to Edge Functions
- **pg_cron** – Scheduled processing and retries
- **Triggers** – Detect content changes
- **Edge Functions** – Generate embeddings via OpenAI

### Simplified Example
```sql
-- Enable extensions
create extension if not exists vector;
create extension if not exists pgmq;

-- Create queue
select pgmq.create('embedding_jobs');

-- Trigger function to queue embedding jobs
create or replace function queue_embeddings()
returns trigger language plpgsql as $$
begin
    perform pgmq.send(
        'embedding_jobs',
        jsonb_build_object(
            'id', NEW.id,
            'table', TG_TABLE_NAME,
            'content', NEW.content
        )
    );
    return NEW;
end;
$$;

-- Attach trigger to your table
create trigger embed_on_insert
    after insert on documents
    for each row
    execute function queue_embeddings();
```

This setup ensures embeddings are automatically generated and updated whenever your data changes, with built-in retries and failure handling.

## 6. Integration with Streaming Architectures

For real-time AI applications, vector databases can integrate with streaming platforms like Kafka.

### Example: Real-time Product Indexing
```python
from kafka import KafkaConsumer
from sentence_transformers import SentenceTransformer
import pinecone
import json

# Initialize
model = SentenceTransformer('all-MiniLM-L6-v2')
pinecone.init(api_key='your-key', environment='us-west1-gcp')
index = pinecone.Index('products')

# Kafka consumer
consumer = KafkaConsumer(
    'product-events',
    bootstrap_servers=['localhost:9092'],
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)

# Process stream
for message in consumer:
    product = message.value
    text = f"{product['name']} {product['description']}"
    embedding = model.encode(text).tolist()
    
    index.upsert(vectors=[(
        product['id'],
        embedding,
        {'name': product['name'], 'category': product['category']}
    )])
```

This pattern enables **real-time semantic search** where products are searchable immediately after addition.

## 8. Pinecone vs. pgvector: Comparison

| Aspect | Pinecone | Supabase pgvector |
|--------|----------|-------------------|
| **Type** | Fully managed service | PostgreSQL extension |
| **Infrastructure** | Zero运维, auto-scaling | You manage PostgreSQL |
| **Pricing** | Pay per usage (stored vectors + queries) | Part of Supabase plan |
| **Integration** | Standalone, API-based | Within PostgreSQL, SQL access |
| **Best For** | Large scale,不想管理 infrastructure | Existing PostgreSQL apps, ACID requirements |
| **Indexing** | Automatic, optimized | You create and maintain indexes |
| **Latency** | Millisecond | Millisecond (with good indexes) |

## 9. Best Practices

### 1. **Choose the Right Chunk Size**
- 500–1000 tokens for most RAG applications
- Overlap 10–20% to maintain context

### 2. **Batch Operations**
```python
# Pinecone - batch upserts
index.upsert(vectors=batch_of_100)  # More efficient than single inserts
```

### 3. **Use Metadata Filtering**
- Filter before vector search when possible
- Combine with vector search for precise results

### 4. **Monitor Index Statistics**
```python
# Pinecone
stats = index.describe_index_stats()
print(f"Total vectors: {stats['total_vector_count']}")
```

### 5. **Plan for Production**
- Enable deletion protection for critical indexes
- Implement retry logic for API calls
- Cache frequent queries

### 6. **Clean Up Resources**
```python
# Delete index when done (saves costs)
pc.delete_index(index_name)
```
