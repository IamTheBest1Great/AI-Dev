# 📘 RAG Basics — Upload Doc → Embed → Store → Query

**Retrieval-Augmented Generation (RAG)** is the most important pattern for building AI applications that use your own data. It combines the power of large language models (LLMs) with a retrieval system that fetches relevant information from your documents.

This guide walks you through the **complete RAG pipeline** — from uploading a document to answering questions about it — with concrete code examples you can use in your MERN stack applications.

---

## 1. What is RAG and Why Do You Need It?

Large language models like GPT-4 are trained on public data up to a certain date. They don't know about your private documents, recent information, or niche domain knowledge. RAG solves this by:

- **Retrieving** relevant pieces of information from your documents based on a user's question.
- **Augmenting** the prompt with that retrieved context.
- **Generating** a final answer using the LLM.

**Benefits over fine-tuning**:
- No training required — works with any LLM.
- Easy to update — just add/remove documents.
- Transparent — you can see which documents were used.
- Reduces hallucinations by grounding the LLM in facts.

---

## 2. High-Level RAG Workflow

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  Upload  │ -> │  Chunk   │ -> │  Embed   │ -> │  Store   │ -> │  Query   │
│ Document │    │          │    │          │    │          │    │          │
└──────────┘    └──────────┘    └──────────┘    └──────────┘    └──────────┘
                                                                        │
                                                                        v
                                                               ┌─────────────────┐
                                                               │  Generate Answer│
                                                               │  (LLM + Context)│
                                                               └─────────────────┘
```

**Steps**:
1. **Upload** – User provides a file (PDF, DOCX, TXT, etc.).
2. **Chunk** – Split the document into smaller, meaningful pieces.
3. **Embed** – Convert each chunk into a vector using an embedding model.
4. **Store** – Save vectors and metadata in a vector database (Pinecone, pgvector).
5. **Query** – User asks a question → embed it → retrieve most similar chunks.
6. **Generate** – Feed retrieved chunks + question to an LLM to produce an answer.

## 3. Detailed Steps with Code Examples

We'll build a simple **PDF Q&A bot** using:
- **OpenAI** for embeddings and chat completion.
- **Pinecone** as the vector database (or you can substitute pgvector).
- **LangChain.js** for orchestration (optional, but recommended).
- **PDF parsing** with `pdf-parse` (Node.js) or similar.

### 3.2 Step 1: Upload and Parse Document

For this example, we'll accept a PDF file. We'll use `pdf-parse` to extract text.

```javascript
import fs from 'fs';
import pdf from 'pdf-parse';

async function parsePDF(filePath) {
  const dataBuffer = fs.readFileSync(filePath);
  const data = await pdf(dataBuffer);
  return data.text; // extracted text
}

// Usage
const text = await parsePDF('./my-document.pdf');
```

**Handling other file types**:
- **TXT**: `fs.readFileSync(path, 'utf-8')`
- **DOCX**: Use `mammoth` or `docx` library.
- **HTML**: Use `cheerio` to extract text.

### 3.3 Step 2: Chunk the Text

LLMs have context windows, and retrieval works best with small, focused chunks. Common strategies:

- **Fixed-size chunking** with overlap (simplest).
- **Recursive character text splitting** (LangChain's default).
- **Semantic chunking** (by paragraphs, sentences, or using embeddings).

We'll use LangChain's `RecursiveCharacterTextSplitter`:

```javascript
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';

async function chunkText(text) {
  const splitter = new RecursiveCharacterTextSplitter({
    chunkSize: 1000,        // number of characters per chunk
    chunkOverlap: 200,      // overlap between chunks
  });
  const chunks = await splitter.createDocuments([text]);
  return chunks.map(doc => doc.pageContent);
}

const chunks = await chunkText(text);
console.log(`Created ${chunks.length} chunks.`);
```

**Why chunk?**
- Embedding models have token limits (e.g., 8k tokens).
- Smaller chunks improve retrieval precision.
- Overlap prevents cutting off important context at boundaries.

### 3.4 Step 3: Embed Each Chunk

Use OpenAI's embedding API to convert each chunk into a vector.

```javascript
import OpenAI from 'openai';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

async function embedChunks(chunks) {
  const embeddings = [];
  // Process in batches to avoid rate limits
  for (let i = 0; i < chunks.length; i += 20) {
    const batch = chunks.slice(i, i + 20);
    const response = await openai.embeddings.create({
      model: 'text-embedding-3-small',
      input: batch,
    });
    embeddings.push(...response.data.map(item => item.embedding));
  }
  return embeddings;
}

const vectors = await embedChunks(chunks);
```

### 3.5 Step 4: Store in Pinecone

We'll store each chunk as a vector with metadata (including the original text for retrieval).

```javascript
import { Pinecone } from '@pinecone-database/pinecone';

const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index(process.env.PINECONE_INDEX);

async function storeVectors(chunks, vectors, metadata = {}) {
  const records = chunks.map((chunk, i) => ({
    id: `chunk-${Date.now()}-${i}`, // unique ID
    values: vectors[i],
    metadata: {
      text: chunk,
      source: metadata.source || 'unknown',
      ...metadata
    }
  }));

  // Upsert in batches of 100
  for (let i = 0; i < records.length; i += 100) {
    const batch = records.slice(i, i + 100);
    await index.upsert(batch);
  }
  console.log(`Stored ${records.length} vectors.`);
}

await storeVectors(chunks, vectors, { source: 'my-document.pdf' });
```

### 3.6 Step 5: Query – Embed User Question and Retrieve

When a user asks a question:
1. Embed the question using the same embedding model.
2. Query Pinecone for the most similar vectors.
3. Return the top-k chunks (with their text).

```javascript
async function retrieveRelevantChunks(question, topK = 5) {
  // 1. Embed question
  const response = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: question,
  });
  const questionEmbedding = response.data[0].embedding;

  // 2. Query Pinecone
  const queryResult = await index.query({
    vector: questionEmbedding,
    topK,
    includeMetadata: true,
  });

  // 3. Extract text from metadata
  return queryResult.matches.map(match => ({
    text: match.metadata.text,
    score: match.score,
  }));
}

// Example
const results = await retrieveRelevantChunks('When was the Eiffel Tower built?');
results.forEach(r => console.log(`Score: ${r.score}\nText: ${r.text}\n`));
```

### 3.7 Step 6: Generate Answer with LLM

Now we have the most relevant chunks. We'll combine them into a context and ask the LLM to answer based solely on that context.

```javascript
async function generateAnswer(question, relevantChunks) {
  const context = relevantChunks.map(c => c.text).join('\n\n---\n\n');
  
  const completion = await openai.chat.completions.create({
    model: 'gpt-4o-mini', // or 'gpt-3.5-turbo'
    messages: [
      { 
        role: 'system', 
        content: 'You are a helpful assistant. Answer the question based only on the provided context. If the context does not contain the answer, say "I cannot find that information in the documents."' 
      },
      { role: 'user', content: `Context:\n${context}\n\nQuestion: ${question}` }
    ],
    temperature: 0, // low temperature for factual answers
  });

  return completion.choices[0].message.content;
}

const answer = await generateAnswer('When was the Eiffel Tower built?', results);
console.log('Answer:', answer);
```

## 4. Complete RAG Pipeline (Putting It All Together)

Here's a unified function that processes a PDF and sets up a queryable index:

```javascript
async function indexDocument(filePath, metadata) {
  // 1. Parse
  const text = await parsePDF(filePath);
  
  // 2. Chunk
  const chunks = await chunkText(text);
  
  // 3. Embed
  const vectors = await embedChunks(chunks);
  
  // 4. Store
  await storeVectors(chunks, vectors, metadata);
  
  console.log('Document indexed successfully!');
}

async function ask(question) {
  // 1. Retrieve
  const relevantChunks = await retrieveRelevantChunks(question);
  
  // 2. Generate
  const answer = await generateAnswer(question, relevantChunks);
  
  return { answer, sources: relevantChunks };
}

// Usage
await indexDocument('./eiffel-tower.pdf', { source: 'eiffel-tower.pdf' });
const response = await ask('Who designed the Eiffel Tower?');
console.log(response.answer);
```

## 5. Using LangChain.js (Simplified)

LangChain provides abstractions that make RAG even easier. Here's the same pipeline with LangChain:

```javascript
import { PDFLoader } from 'langchain/document_loaders/fs/pdf';
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';
import { OpenAIEmbeddings } from '@langchain/openai';
import { PineconeStore } from '@langchain/pinecone';
import { Pinecone } from '@pinecone-database/pinecone';
import { ChatOpenAI } from '@langchain/openai';
import { createRetrievalChain } from 'langchain/chains/retrieval';
import { createStuffDocumentsChain } from 'langchain/chains/combine_documents';
import { ChatPromptTemplate } from '@langchain/core/prompts';

// 1. Load and split
const loader = new PDFLoader('./eiffel-tower.pdf');
const docs = await loader.load();
const splitter = new RecursiveCharacterTextSplitter({ chunkSize: 1000, chunkOverlap: 200 });
const splitDocs = await splitter.splitDocuments(docs);

// 2. Create vector store
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index(process.env.PINECONE_INDEX);
const embeddings = new OpenAIEmbeddings({ model: 'text-embedding-3-small' });
const vectorStore = await PineconeStore.fromDocuments(splitDocs, embeddings, { pineconeIndex: index });

// 3. Build retrieval chain
const retriever = vectorStore.asRetriever({ k: 5 });
const llm = new ChatOpenAI({ model: 'gpt-4o-mini', temperature: 0 });

const prompt = ChatPromptTemplate.fromTemplate(`
Answer the question based only on the following context:
{context}

Question: {input}
`);

const documentChain = await createStuffDocumentsChain({ llm, prompt });
const retrievalChain = await createRetrievalChain({ retriever, combineDocsChain: documentChain });

// 4. Ask
const result = await retrievalChain.invoke({ input: 'When was the Eiffel Tower built?' });
console.log(result.answer);
```

LangChain handles the retrieval and document combination automatically.

## 6. Best Practices and Tips

### Chunking Strategy
- **Size**: 500–1500 tokens (roughly 2000–6000 characters) works well for most RAG applications.
- **Overlap**: 10–20% of chunk size.
- **Method**: For general text, recursive character splitting by `\n\n`, `\n`, space is effective. For code, use language‑specific splitters.

### Metadata
- Store the original text in metadata so you don't need to retrieve it separately.
- Include source filename, page numbers, timestamps for filtering.
- In Pinecone, metadata is searchable; in pgvector, use JSONB columns and GIN indexes for filtering.

### Retrieval Tuning
- Experiment with `topK` – too few may miss context, too many may include noise.
- Set a similarity threshold (e.g., only consider chunks with score > 0.7).
- Consider **hybrid search** (vector + keyword) for better results on exact matches.

### Prompt Engineering
- Clearly instruct the LLM to use only the provided context.
- If no relevant context, instruct to say "I don't know" to avoid hallucinations.
- Include the original question and context in a structured format.

### Handling Large Documents
- For very large collections, use a more efficient index (HNSW in pgvector, larger pods in Pinecone).
- Consider **hierarchical retrieval**: first retrieve relevant documents, then chunks within them.

## 7. Common Pitfalls and Solutions

| Problem | Solution |
|---------|----------|
| **No relevant chunks retrieved** | Check chunk size; maybe chunks are too large or too small. Try increasing `topK`. |
| **LLM ignores context** | Strengthen prompt: "Answer based ONLY on the context. If not in context, say so." |
| **High latency** | Use smaller embedding model, reduce `topK`, optimize vector index. |
| **Costs too high** | Cache frequent queries, batch embedding calls, use cheaper models for non‑critical tasks. |
| **Poor answer quality** | Improve chunking, add reranking step (cross‑encoder) to reorder retrieved chunks. |

## 8. Next Steps: Advanced RAG

Once you master basic RAG, you can explore:
- **Advanced chunking** (semantic splitting, agentic chunking).
- **Re-ranking** using cross‑encoders (e.g., Cohere rerank).
- **Hybrid search** (combine vector with BM25/full‑text).
- **Multi‑query retrieval** – generate multiple questions from the user query to cover more angles.
- **Self‑querying retrievers** – use LLM to extract filters from the question.
- **Graph RAG** – incorporate knowledge graphs.
