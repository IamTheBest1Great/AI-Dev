## 1. What is a Knowledge Graph?

A **knowledge graph** structures information as a network of **nodes** (entities) connected by **relationships**. Instead of isolated text chunks, it captures how things are connected.

**Example**:
```
(Paul Atreides) -[:HEIR_OF]-> (House Atreides) -[:RULES]-> (Caladan)
```
This shows that Paul is heir to House Atreides, which rules the planet Caladan—a relationship pure vector search would miss, but a graph traversal finds instantly.

### Why Ground AI with Knowledge Graphs?

Unlike vector‑only RAG, **GraphRAG** combines semantic search with structured relationships. Key benefits:

- **Multi‑hop reasoning** – Answer questions that require traversing several relationships (“Which planets are ruled by families Paul is heir to?”).
- **Explainability** – Show the exact path of relationships used to derive an answer.
- **Precision** – Eliminate false positives by following explicit connections, not just similarity scores.
- **Dynamic updates** – Add new entities and relationships in real‑time as your data evolves.
- **Global structure** – Understand how concepts relate *across* documents, not just within one chunk.

## 2. Neo4j Basics

Neo4j is the leading graph database. It stores **property graphs**—nodes, relationships, and key‑value properties—and uses the **Cypher** query language.

### 2.1 Data Model

| Component | Description | Example |
|-----------|-------------|---------|
| **Node** | An entity | `(:Person {name: 'Paul'})` |
| **Relationship** | Directed connection between two nodes | `(Paul)-[:HEIR_OF]->(HouseAtreides)` |
| **Property** | Key‑value pairs on nodes or relationships | `{age: 15, title: 'Duke'}` |
| **Label** | Groups nodes by role (like a table name) | `:Person`, `:House`, `:Planet` |

**Why it matters for developers**: Graph databases traverse relationships in constant time, avoiding complex SQL JOINs. For AI apps, this means you can instantly find all concepts related to a user’s question.

### 2.2 Cypher Query Language

Cypher is Neo4j’s declarative language (like SQL for graphs). Key patterns:

| Pattern | Cypher example | What it does |
|---------|---------------|--------------|
| Create node | `CREATE (p:Person {name: 'Alice'})` | Inserts a person node |
| Create relationship | `MATCH (a:Person), (b:Company) WHERE a.name = 'Alice' AND b.name = 'Neo4j' CREATE (a)-[:WORKS_AT]->(b)` | Connects Alice to Neo4j |
| Simple traversal | `MATCH (p:Person)-[:KNOWS]->(friend) RETURN friend.name` | Finds friends of a person |
| Multi‑hop | `MATCH (p:Person)-[:PURCHASED]->(:Product)-[:CATEGORY]->(cat) RETURN cat.name` | Which product categories did this person buy from? |
| Path finding | `MATCH path = shortestPath((a)-[*]-(b)) RETURN path` | Shortest connection between two nodes |

**AI relevance**: You can translate a user’s natural language question into a Cypher query using an LLM (e.g., LangChain’s `GraphCypherQAChain`), enabling complex reasoning over your data.

## 3. Building a Knowledge Graph from Documents (LLM‑Extracted Entities)

Manually building a knowledge graph is slow. Modern pipelines use LLMs to automatically extract entities and relationships from unstructured text, then insert them into Neo4j.

The **core pattern**:
1. Feed text to an LLM with a prompt: “Extract all entities and relationships as JSON.”
2. Define a strict schema (e.g., with Zod) to ensure valid output.
3. Parse, validate, and batch‑insert into Neo4j using `MERGE` to avoid duplicates.

### 3.1 Installation (neo4j‑graphrag)

```bash
pip install "neo4j-graphrag[openai]"
```

### 3.2 Define Your Schema

First, tell the LLM what entity types and relationships to look for.

```python
from neo4j import GraphDatabase
from neo4j_graphrag.llm import OpenAILLM
from neo4j_graphrag.embeddings import OpenAIEmbeddings
from neo4j_graphrag.experimental.pipeline.kg_builder import SimpleKGPipeline

# Connect to Neo4j
driver = GraphDatabase.driver(
    "neo4j://localhost:7687",
    auth=("neo4j", "password")
)

# Define schema
node_types = ["Person", "House", "Planet"]
relationship_types = ["PARENT_OF", "HEIR_OF", "RULES"]
patterns = [
    ("Person", "PARENT_OF", "Person"),
    ("Person", "HEIR_OF", "House"),
    ("House", "RULES", "Planet"),
]

# Initialize LLM and embedder
llm = OpenAILLM(
    model_name="gpt-4o",
    model_params={
        "max_tokens": 2000,
        "response_format": {"type": "json_object"},
        "temperature": 0,
    }
)
embedder = OpenAIEmbeddings(model="text-embedding-3-large")
```

### 3.3 Build the Graph Automatically

```python
# Initialize the pipeline
kg_builder = SimpleKGPipeline(
    llm=llm,
    driver=driver,
    embedder=embedder,
    schema={
        "node_types": node_types,
        "relationship_types": relationship_types,
        "patterns": patterns,
    },
    from_pdf=False,  # Set to True for PDF files
)

# Process a text document
text = """
The son of Duke Leto Atreides and the Lady Jessica, Paul is the heir of House 
Atreides, an aristocratic family that rules the planet Caladan.
"""

import asyncio
asyncio.run(kg_builder.run_async(text=text))
```

The pipeline automatically:
- Extracts entities and relationships using the LLM
- Creates nodes with appropriate labels
- Establishes relationships between nodes
- Generates embeddings for vector search

### 3.4 Advanced: Custom Entity Extraction (Unstructured.io)

For richer document parsing (tables, images, headings), use Unstructured.io:

```python
from unstructured.partition.auto import partition

# Parse document with layout awareness
```

### 3.5 Relationship Mapping & Entity Resolution

To maintain quality as the graph grows:

- **Canonicalization**: Merge “Elon Musk”, “Elon”, “Musk” into one node. Use exact matching on normalized names or compare embeddings.
- **Confidence scoring**: Ask the LLM to output a confidence score (0–1) for each relationship. Filter low‑confidence edges before insertion.
- **Incremental updates**: When new documents arrive, run extraction again and use `MERGE`—do not rebuild the whole graph. This keeps the graph live.

**MERN integration tip**: Store the raw extraction job in MongoDB (status, source document), run extraction in a BullMQ job, and write results to Neo4j. This separates concerns and allows retries.

## 4. GraphRAG: Querying Knowledge Graphs

Once the knowledge graph is built, you retrieve information using **hybrid approaches** that combine vector similarity and graph traversal.

### 4.1 Vector + Graph Hybrid Retrieval

The most powerful pattern: start with a vector search, then expand via graph relationships.

```python
from neo4j_graphrag.retrievers import HybridCypherRetriever
from neo4j_graphrag.embeddings import OpenAIEmbeddings
from neo4j_graphrag.generation import GraphRAG
from neo4j_graphrag.llm import OpenAILLM

# Initialize
embedder = OpenAIEmbeddings(model="text-embedding-3-large")
llm = OpenAILLM(model_name="gpt-4o", model_params={"temperature": 0})

# Retrieval query that traverses relationships
retrieval_query = """
WITH node, score
OPTIONAL MATCH (node)-[:NEXT_CHUNK]-(neighbor)  // Get adjacent chunks
OPTIONAL MATCH (node)<-[HAS_ENTITY]-(entity)     // Get linked entities
ORDER BY score DESC LIMIT 100
RETURN apoc.convert.toSet(
    COLLECT(elementId(node)) + 
    COLLECT(elementId(entity)) + 
    COLLECT(elementId(neighbor))
) AS listIds,
COLLECT (entity.id) as contextNodes,
node.text as nodeText,
score
ORDER BY score DESC
"""

# Create hybrid retriever
retriever = HybridCypherRetriever(
    driver,
    vector_index_name="document_chunks",
    fulltext_index_name="entity_terms",
    retrieval_query=retrieval_query,
    embedder=embedder,
)

# Build RAG pipeline
rag = GraphRAG(retriever=retriever, llm=llm)

# Query
response = rag.search(
    "What planets are connected to House Atreides?",
    retriever_config={"top_k": 5}
)
print(response.answer)
```

This discovers **semantically similar chunks**, plus:
- Entities extracted from those chunks
- Neighboring chunks in the document sequence
- Related content like tables and images

### 4.2 Multi‑Hop Reasoning

For questions that require multiple steps, you can use targeted Cypher queries.

```python
def multi_hop_reasoning(query_text):
    # Step 1: Extract entities from query
    entities = extract_entities_with_llm(query_text)
    
    # Step 2: Traverse relationships (up to 3 hops)
    with driver.session() as session:
        result = session.run(
            """
            MATCH path = (start:Entity)-[:RELATED_TO*1..3]-(end:Entity)
            WHERE start.name IN $entities
            RETURN path, 
                   [node IN nodes(path) | node.name] AS entity_chain,
                   [rel IN relationships(path) | type(rel)] AS relationship_chain
            LIMIT 10
            """,
            entities=entities
        )
        paths = list(result)
    
    # Step 3: Format paths as context for the LLM
    context = format_paths_for_context(paths)
    
    # Step 4: Generate answer with relationship context
    return generate_answer(query_text, context)
```

**Classic multi‑hop example**: “What startup did the ex‑CEO of our competitor found?”
- Hop 1: Find competitor → CEO relationship → Person.
- Hop 2: Person → `FOUNDED` relationship → Startup.
Vector search alone fails because no single chunk contains both facts.

### 4.4 GraphRAG: Community Summarization & Global Structure

Microsoft’s GraphRAG pattern builds a knowledge graph from all documents, discovers **communities** of related nodes, and summarizes each community. When a question arrives, these community summaries provide global context—going beyond the local chunks. In your own stack, you can combine:

1. Graph built via LLM extraction.
2. At query time, vector search retrieves relevant chunks, and a graph query retrieves relationships (as above).
3. Merge both contexts for a truly holistic answer.

This captures **global structure** (how concepts connect across documents), not just local chunk similarity.

## 6. Advanced GraphRAG Patterns

### 6.1 Lexical Graph Construction

Preserve the document’s own structure (headings, tables, images, chunk order) as a graph:

```cypher
(:Document {name: 'report.pdf'})
  -[:HAS_CHUNK]->(:Chunk {text: '...', embedding: [...]})
  -[:NEXT_CHUNK]->(:Chunk {text: '...'})
(:Chunk)-[:RELATED_CONTENT]->(:Image {url: 'diagram.png'})
(:Chunk)-[:RELATED_CONTENT]->(:Table {data: '...'})
(:Chunk)<-[:HAS_ENTITY]-(:Entity {name: 'Caladan'})
```

Now retrieval understands document flow and can return images alongside text.

### 6.2 Entity Linking for Query Enrichment

When a query comes in, link its entities to the graph to expand context.

```python
def enrich_query_with_graph(query):
    entities = extract_entities(query)
    with driver.session() as session:
        result = session.run(
            """
            MATCH (e:Entity)-[r]-(related)
            WHERE e.name IN $entities
            RETURN e.name, type(r), related.name
            LIMIT 5
            """,
            entities=entities
        )
        relationships = list(result)
    
    enriched = f"{query}\n\nRelated information: "
    for rel in relationships:
        enriched += f"{rel['e.name']} {rel['type(r)']} {rel['related.name']}; "
    return enriched
```

### 6.3 Reasoning Layer Architecture

For production systems, implement a layered approach:
1. **Entity Extraction** – Identify key entities in the user query.
2. **Graph Traversal** – Follow relationships to find connected information.
3. **Path Ranking** – Score relationship paths by relevance.
4. **Context Assembly** – Format discovered paths as structured context.
5. **Reasoned Generation** – Feed both documents and relationship paths to the LLM.

## 8. Best Practices

### 8.1 Schema Design
- Use meaningful labels: `:Person`, not `:Node`.
- Create indexes on frequently queried properties.
- Model domain rules explicitly with relationship types.

### 8.2 Performance Optimization
```cypher
// Indexes for fast lookups
CREATE INDEX person_name FOR (n:Person) ON (n.name);
CREATE INDEX entity_type FOR (n:Entity) ON (n.type);

// Vector index for embeddings
CREATE VECTOR INDEX document_embeddings IF NOT EXISTS
FOR (n:Chunk) ON (n.embedding)
OPTIONS {indexConfig: {
  `vector.dimensions`: 1536,
  `vector.similarity_function`: 'cosine'
}};
```

- Limit traversal depth in multi‑hop queries.
- Tune vector vs. keyword weights in hybrid search.
- Use `MERGE` to avoid duplicate nodes and relationships on insertion.
- Keep tool outputs concise to avoid token waste.

### 8.3 Monitoring
Track: query latency (especially multi‑hop), entity extraction accuracy, graph growth rate, cache hit ratios.

## 9. Common Pitfalls and Solutions

| Problem | Solution |
|---------|----------|
| **Slow multi‑hop queries** | Limit traversal depth, use indexes |
| **Entity extraction misses relationships** | Refine LLM prompts, provide few‑shot examples |
| **Graph becomes too dense** | Prune low‑confidence relationships |
| **Hybrid search returns irrelevant results** | Tune vector vs. keyword weights, add metadata filtering |
| **Memory consumption grows** | Archive old conversation nodes |
| **Duplicate nodes (“Elon” vs. “Elon Musk”)** | Implement canonicalization and merging logic |
| **Stale data** | Design incremental updates; use event‑driven pipelines |
