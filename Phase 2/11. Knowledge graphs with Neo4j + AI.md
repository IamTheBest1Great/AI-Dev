# 📘 Knowledge Graphs with Neo4j + AI — Complete Guide

Knowledge graphs represent a paradigm shift in how we structure information for AI applications. Unlike vector databases that treat documents as isolated chunks, knowledge graphs capture **entities** (people, places, concepts) and the **relationships** between them, enabling true reasoning and multi-hop understanding.

This guide shows you how to combine Neo4j—the leading graph database—with AI to build intelligent systems that don't just retrieve but actually **understand** your data.

---

## 1. What is a Knowledge Graph?

A knowledge graph organizes information as a network of **nodes** (entities) connected by **relationships**. For example:

```
(Paul Atreides) -[:HEIR_OF]-> (House Atreides) -[:RULES]-> (Caladan)
```

This structure captures that Paul is heir to House Atreides, which rules the planet Caladan—a relationship that pure vector search would miss .

### Why Ground AI with Knowledge Graphs?

Unlike vector-only RAG, GraphRAG combines semantic search with structured relationships to deliver more accurate and contextually relevant responses . Key benefits:

- **Multi-hop reasoning** – Answer questions that require traversing relationships ("What planets are ruled by families Paul is heir to?")
- **Explainability** – Show the exact path of relationships used to derive an answer 
- **Precision** – Eliminate false positives by following explicit connections rather than just similarity scores
- **Dynamic updates** – Add new entities and relationships in real-time as your data evolves

## 2. Core Concepts in Neo4j

### 2.1 Data Model

Neo4j uses a property graph model with these components:

| Component | Description | Example |
|-----------|-------------|---------|
| **Node** | Represents an entity | `(:Person {name: 'Paul'})` |
| **Relationship** | Connects two nodes with direction and type | `(Paul)-[:HEIR_OF]->(HouseAtreides)` |
| **Property** | Key-value pairs on nodes or relationships | `{age: 15, title: 'Duke'}` |
| **Label** | Groups nodes by role | `:Person`, `:House`, `:Planet` |

### 2.2 Cypher Query Language

Cypher is Neo4j's intuitive query language, designed to express graph patterns:

```cypher
// Find all planets ruled by houses that Paul is heir to
MATCH (p:Person {name: 'Paul'})-[:HEIR_OF]->(h:House)-[:RULES]->(planet:Planet)
RETURN planet.name
```

This pattern matching is what makes graph databases so powerful for connected data.

## 3. Building a Knowledge Graph from Documents

Modern knowledge graph construction uses LLMs to extract entities and relationships from unstructured text. The `neo4j-graphrag` Python package provides official tools for this .

### 3.1 Installation

```bash
pip install "neo4j-graphrag[openai]"
```

### 3.2 Define Your Schema

First, define what entities and relationships the LLM should look for:

```python
from neo4j import GraphDatabase
from neo4j_graphrag.llm import OpenAILLM
from neo4j_graphrag.embeddings import OpenAIEmbeddings
from neo4j_graphrag.experimental.pipeline.kg_builder import SimpleKGPipeline

# Connect to Neo4j
driver = GraphDatabase.driver(
    "neo4j://localhost:7687",
    auth=("neo4j", "password")
)

# Define schema
node_types = ["Person", "House", "Planet"]
relationship_types = ["PARENT_OF", "HEIR_OF", "RULES"]
patterns = [
    ("Person", "PARENT_OF", "Person"),
    ("Person", "HEIR_OF", "House"),
    ("House", "RULES", "Planet"),
]

# Initialize LLM and embedder
llm = OpenAILLM(
    model_name="gpt-4o",
    model_params={
        "max_tokens": 2000,
        "response_format": {"type": "json_object"},
        "temperature": 0,
    }
)
embedder = OpenAIEmbeddings(model="text-embedding-3-large")
```

### 3.3 Build the Graph

```python
# Initialize the pipeline
kg_builder = SimpleKGPipeline(
    llm=llm,
    driver=driver,
    embedder=embedder,
    schema={
        "node_types": node_types,
        "relationship_types": relationship_types,
        "patterns": patterns,
    },
    from_pdf=False,  # Set to True for PDF files
)

# Process a text document
text = """
The son of Duke Leto Atreides and the Lady Jessica, Paul is the heir of House 
Atreides, an aristocratic family that rules the planet Caladan.
"""

import asyncio
asyncio.run(kg_builder.run_async(text=text))
```

The pipeline automatically:
- Extracts entities and relationships using the LLM
- Creates nodes with appropriate labels
- Establishes relationships between nodes
- Generates embeddings for vector search 

### 3.4 Advanced: Custom Entity Extraction

For better control, you can use Unstructured.io for document parsing, which handles complex elements like tables and images :

## 4. GraphRAG: Querying Knowledge Graphs

Once your knowledge graph is built, you need to retrieve information from it. Neo4j offers several retrieval patterns.

### 4.1 Vector + Graph Hybrid Retrieval

The most powerful approach combines vector similarity with graph traversal :

```python
from neo4j_graphrag.retrievers import HybridCypherRetriever
from neo4j_graphrag.embeddings import OpenAIEmbeddings
from neo4j_graphrag.generation import GraphRAG
from neo4j_graphrag.llm import OpenAILLM

# Initialize components
embedder = OpenAIEmbeddings(model="text-embedding-3-large")
llm = OpenAILLM(model_name="gpt-4o", model_params={"temperature": 0})

# Define retrieval query that traverses relationships
retrieval_query = """
WITH node, score
OPTIONAL MATCH (node)-[:NEXT_CHUNK]-(neighbor)  // Get adjacent chunks
OPTIONAL MATCH (node)<-[HAS_ENTITY]-(entity)     // Get linked entities
ORDER BY score DESC LIMIT 100
RETURN apoc.convert.toSet(
    COLLECT(elementId(node)) + 
    COLLECT(elementId(entity)) + 
    COLLECT(elementId(neighbor))
) AS listIds,
COLLECT (entity.id) as contextNodes,
node.text as nodeText,
score
ORDER BY score DESC
"""

# Create hybrid retriever
retriever = HybridCypherRetriever(
    driver,
    vector_index_name="document_chunks",
    fulltext_index_name="entity_terms",
    retrieval_query=retrieval_query,
    embedder=embedder,
)

# Build RAG pipeline
rag = GraphRAG(retriever=retriever, llm=llm)

# Query
response = rag.search(
    "What planets are connected to House Atreides?",
    retriever_config={"top_k": 5}
)
print(response.answer)
```

This hybrid approach discovers not just semantically similar chunks, but also:
- Entities extracted from those chunks
- Neighboring chunks in the document sequence
- Related content like tables and images 

### 4.2 Multi-Hop Reasoning

For complex questions requiring relationship traversal, you can use targeted Cypher queries :

```python
def multi_hop_reasoning(query_text):
    # Step 1: Extract entities from query
    entities = extract_entities_with_llm(query_text)
    
    # Step 2: Traverse relationships
    with driver.session() as session:
        result = session.run(
            """
            MATCH path = (start:Entity)-[:RELATED_TO*1..3]-(end:Entity)
            WHERE start.name IN $entities
            RETURN path, 
                   [node IN nodes(path) | node.name] AS entity_chain,
                   [rel IN relationships(path) | type(rel)] AS relationship_chain
            LIMIT 10
            """,
            entities=entities
        )
        paths = list(result)
    
    # Step 3: Format paths for LLM
    context = format_paths_for_context(paths)
    
    # Step 4: Generate answer with relationship context
    return generate_answer(query_text, context)
```

## 5. Persisting Conversation Memory

Knowledge graphs excel at storing conversational context across sessions :

```python
from langchain.memory import Neo4jGraphMemory

# Initialize graph memory
memory = Neo4jGraphMemory(
    graph_db=graph,
    user_id="user_123",
    conversation_id="conv_456",
    context_window=5  # Keep last 5 exchanges
)

# Store conversation turns
memory.save_context(
    {"input": "Tell me about House Atreides"},
    {"output": "House Atreides rules Caladan and values honor."}
)

# Retrieve context for new query
context = memory.load_memory_variables({})
```

The graph structure enables powerful memory features:
- Cross-session user preference tracking
- Topic-based conversation clustering
- Relationship discovery between conversation topics

## 6. Advanced GraphRAG Patterns

### 6.1 Lexical Graph Construction

For technical documents, build a **lexical graph** that preserves document structure :

```cypher
// Create document hierarchy
(:Document {name: 'report.pdf'})
  -[:HAS_CHUNK]->(:Chunk {text: '...', embedding: [...]})
  -[:NEXT_CHUNK]->(:Chunk {text: '...'})
  
// Link to extracted content
(:Chunk)-[:RELATED_CONTENT]->(:Image {url: 'diagram.png'})
(:Chunk)-[:RELATED_CONTENT]->(:Table {data: '...'})

// Connect entities
(:Chunk)<-[:HAS_ENTITY]-(:Entity {name: 'Caladan'})
```

This structure allows retrieval that understands document flow and can return images alongside text.

### 6.2 Entity Linking for Query Enrichment

When a query comes in, link it to graph entities to expand context :

```python
def enrich_query_with_graph(query):
    # Extract entities
    entities = extract_entities(query)
    
    # Find related entities in graph
    with driver.session() as session:
        result = session.run(
            """
            MATCH (e:Entity)-[r]-(related)
            WHERE e.name IN $entities
            RETURN e.name, type(r), related.name
            LIMIT 5
            """,
            entities=entities
        )
        relationships = list(result)
    
    # Enrich original query
    enriched = f"{query}\n\nRelated information: "
    for rel in relationships:
        enriched += f"{rel['e.name']} {rel['type(r)']} {rel['related.name']}; "
    
    return enriched
```

### 6.3 Reasoning Layer Architecture

For production systems, implement a reasoning layer that sits between retrieval and generation :

1. **Entity Extraction** – Identify key entities in the user query
2. **Graph Traversal** – Follow relationships to find connected information
3. **Path Ranking** – Score relationship paths by relevance
4. **Context Assembly** – Format discovered paths as structured context
5. **Reasoned Generation** – Feed both documents and relationship paths to LLM

## 8. Best Practices

### 8.1 Schema Design

- **Use meaningful labels** – `:Person` not `:Node`
- **Create indexes** – On frequently queried properties
- **Model domains explicitly** – Capture business logic in relationships
- **Avoid over-normalization** – Graphs thrive on connections

### 8.2 Performance Optimization

```cypher
// Create indexes for fast lookups
CREATE INDEX person_name FOR (n:Person) ON (n.name);
CREATE INDEX entity_type FOR (n:Entity) ON (n.type);

// Create vector index for embeddings
CREATE VECTOR INDEX document_embeddings IF NOT EXISTS
FOR (n:Chunk) ON (n.embedding)
OPTIONS {indexConfig: {
  `vector.dimensions`: 1536,
  `vector.similarity_function`: 'cosine'
}};
```

### 8.3 Monitoring

Track these metrics:
- Query latency (especially for multi-hop traversals)
- Entity extraction accuracy
- Graph growth rate
- Cache hit ratios for frequent queries

## 9. Common Pitfalls and Solutions

| Problem | Solution |
|---------|----------|
| **Slow multi-hop queries** | Limit traversal depth, use indexes |
| **Entity extraction misses relationships** | Refine LLM prompts, provide examples |
| **Graph becomes too dense** | Prune low-confidence relationships |
| **Hybrid search returns irrelevant results** | Tune vector vs. keyword weights |
| **Memory consumption grows** | Archive old conversation nodes |
