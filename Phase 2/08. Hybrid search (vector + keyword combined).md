# 📘 Hybrid Search (Vector + Keyword Combined) — Complete Guide

Hybrid search combines **semantic (vector) search** with **keyword (lexical) search** to give you the best of both worlds. It’s a core advanced technique for building robust retrieval systems in AI applications.

---

## 1. What Is Hybrid Search?

- **Vector search** uses embeddings to find documents by meaning (synonyms, concepts, intent).  
- **Keyword search** finds exact or partial term matches (BM25, full‑text search).  

Hybrid search merges results from both methods, often with a weighted score, to return more relevant documents than either method alone.

---

## 2. Why Hybrid Search?

Pure vector search can fail when:
- Queries contain rare terms, proper names, or IDs not well‑represented in training data.
- Exact matches are required (e.g., product codes, legal references).
- The embedding model misinterprets the query (out‑of‑domain).

Pure keyword search fails on:
- Synonyms (“car” vs. “automobile”).
- Polysemy (“bank” as river vs. money).
- Concept‑level queries (“how to treat a cold” should find “remedies for flu”).

Hybrid solves both by leveraging the strengths of each.

---

## 3. How Hybrid Search Works

Typical pipeline:
1. Convert user query into a vector (embedding).
2. Run keyword search (e.g., BM25, PostgreSQL `tsvector`) on the same document corpus.
3. Retrieve top‑k candidates from each method.
4. Combine the scores (normalization + weighting) to produce a final ranked list.
5. Return top‑N results.

**Score combination**:
- **Weighted sum**: `final_score = w₁ · vector_score + w₂ · keyword_score`  
  (requires normalizing scores to a comparable scale, e.g., 0–1).
- **Reciprocal Rank Fusion (RRF)**: combines ranks instead of scores:  
  `score = ∑ 1 / (k + rankᵢ)` for each document's rank in each result set (k is a constant, often 60).

RRF is simple and avoids score normalisation.

## 4. Implementation Options

### 4.1 Pinecone

Pinecone offers two ways to implement hybrid search:

#### A. Integrated Embedding Index (Easiest)
When you create an index with the `embed` configuration, Pinecone can automatically perform hybrid search. You just query with text:

```javascript
const index = pc.index('my-embeddings');
const results = await index.search({
  namespace: 'my-namespace',
  query: {
    topK: 10,
    inputs: { text: 'user query' }   // Pinecone handles embedding + hybrid
  }
});
```

#### B. Sparse‑Dense Vectors (Manual)
You store both dense vectors (embeddings) and sparse vectors (e.g., BM25) per document. At query time, you provide both.

**Indexing with sparse vectors**:
```javascript
await index.upsert([{
  id: 'doc1',
  values: denseEmbedding,           // e.g., from OpenAI
  sparseValues: {                    // pre‑computed sparse representation
    indices: [1, 5, 10],              // token IDs
    values: [0.5, 1.2, 0.8]           // BM25 scores or term frequencies
  },
  metadata: { text: '...' }
}]);
```

**Query with sparse + dense**:
```javascript
const queryDense = await getEmbedding(query);
const querySparse = computeBM25Sparse(query);  // token IDs + weights

const results = await index.query({
  vector: queryDense,
  sparseVector: querySparse,
  topK: 10,
  includeMetadata: true
});
```

*Note:* You need a token‑to‑ID mapping for sparse indices, and a way to compute BM25 for the query (using the same corpus statistics).

### 4.2 Supabase pgvector + Full‑Text Search

PostgreSQL offers both full‑text search (via `tsvector`) and vector similarity (via pgvector). You can combine them in SQL.

#### Step 1: Prepare the table
```sql
-- Add a generated tsvector column for full‑text search
ALTER TABLE documents ADD COLUMN fts tsvector
  GENERATED ALWAYS AS (to_tsvector('english', content)) STORED;

CREATE INDEX fts_idx ON documents USING GIN (fts);
CREATE INDEX embedding_idx ON documents USING ivfflat (embedding vector_cosine_ops);
```

#### Step 2: Hybrid search function (RRF method)
```sql
CREATE OR REPLACE FUNCTION hybrid_search_rrf(
  query_text TEXT,
  query_embedding VECTOR(1536),
  match_count INT DEFAULT 10,
  rrf_k INT DEFAULT 60
)
RETURNS TABLE(id BIGINT, content TEXT, rrf_score FLOAT)
LANGUAGE plpgsql
AS $$
DECLARE
  query_tsquery TSQUERY := plainto_tsquery('english', query_text);
BEGIN
  RETURN QUERY
  WITH fulltext AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY ts_rank(fts, query_tsquery) DESC) AS rank
    FROM documents
    WHERE fts @@ query_tsquery
    LIMIT 100
  ),
  vector AS (
    SELECT id, ROW_NUMBER() OVER (ORDER BY embedding <=> query_embedding) AS rank
    FROM documents
    LIMIT 100
  ),
  combined AS (
    SELECT id, rank FROM fulltext
    UNION ALL
    SELECT id, rank FROM vector
  )
  SELECT
    d.id,
    d.content,
    SUM(1.0 / (rrf_k + c.rank)) AS rrf_score
  FROM combined c
  JOIN documents d ON d.id = c.id
  GROUP BY d.id, d.content
  ORDER BY rrf_score DESC
  LIMIT match_count;
END;
$$;
```

**Call from JavaScript**:
```javascript
const { data } = await supabase.rpc('hybrid_search_rrf', {
  query_text: 'user query',
  query_embedding: embeddingArray,
  match_count: 10,
  rrf_k: 60
});
```

#### Step 3: Weighted sum approach (alternative)
If you prefer normalised scores, compute them separately and combine with weights.

## 5. Score Normalization and Weighting

- **Vector similarity**: often cosine (–1 to 1) or normalized to 0–1.
- **Keyword scores**: BM25 can range from 0 to unbounded.
- **Normalization methods**: min‑max scaling, z‑score, or use ranks (RRF).

**Reciprocal Rank Fusion (RRF)** is the simplest: it ignores absolute scores and only needs ranks from each system. It’s robust and widely used.

---

## 6. When to Use Hybrid Search

- Your corpus contains both conceptual content **and** specific terms (names, codes, dates).
- Queries may be out‑of‑domain for your embedding model.
- You want to improve retrieval precision and recall in RAG systems.
- You need exact matches for certain fields (e.g., product IDs) but semantic understanding for others.

---

## 7. Best Practices

- **Test different weights/RRF k** – use an evaluation set to tune.
- **Apply metadata filters early** – reduces the number of candidates for both searches.
- **Cache frequent queries** – hybrid is more expensive, so caching helps.
- **Consider performance** – two queries plus merging adds latency. Parallelise where possible.
- **Monitor result quality** – compare hybrid vs. pure vector on a test set.

## 8. Example: Full RAG with Hybrid Search (Pinecone + LangChain)

```javascript
import { PineconeStore } from '@langchain/pinecone';
import { OpenAIEmbeddings } from '@langchain/openai';
import { ChatOpenAI } from '@langchain/openai';
import { createRetrievalChain } from 'langchain/chains/retrieval';
import { createStuffDocumentsChain } from 'langchain/chains/combine_documents';
import { ChatPromptTemplate } from '@langchain/core/prompts';
import { Pinecone } from '@pinecone-database/pinecone';

// Setup
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index('my-index');
const embeddings = new OpenAIEmbeddings();

// Vector store (dense only)
const vectorStore = await PineconeStore.fromExistingIndex(embeddings, { pineconeIndex: index });

// Create a retriever that uses Pinecone's hybrid search
// (requires Pinecone index created with integrated embeddings)
const retriever = vectorStore.asRetriever({
  k: 10,
  searchType: 'hybrid',  // if supported, else implement custom
});

// Build chain
const llm = new ChatOpenAI({ model: 'gpt-4o-mini' });
const prompt = ChatPromptTemplate.fromTemplate(`
Answer based on context:
{context}
Question: {input}
`);
const documentChain = await createStuffDocumentsChain({ llm, prompt });
const retrievalChain = await createRetrievalChain({ retriever, combineDocsChain: documentChain });

const result = await retrievalChain.invoke({ input: 'What is the capital of France?' });
console.log(result.answer);
```

*Note:* `searchType: 'hybrid'` may not be directly available in LangChain; you might need to extend the retriever to call Pinecone's hybrid query.

- Always test and tune for your specific data and queries.
