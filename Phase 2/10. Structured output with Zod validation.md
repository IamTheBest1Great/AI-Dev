# 📘 Structured Outputs with Zod Validation — Complete Guide

Structured outputs are the key to moving from free‑text LLM responses to **type‑safe, predictable data** that your application can reliably consume. This guide shows you how to combine OpenAI’s JSON mode / function calling with **Zod** – a TypeScript‑first schema validation library – to get exactly the data you need, every time.

---

## 1. What Are Structured Outputs?

When you ask an LLM a question, it typically returns natural language. But for many use cases (extracting data, generating configuration, populating forms), you need **structured data** – JSON objects with known fields, types, and constraints.

**Examples**:
- `{ "name": "John", "age": 30, "city": "New York" }`
- `{ "ingredients": ["flour", "eggs"], "steps": ["mix", "bake"] }`

Structured outputs give you:
- **Predictability** – You know exactly what keys and types to expect.
- **Type safety** – Use the data directly in your code without parsing guesswork.
- **Reduced hallucinations** – Constraining the output format forces the model to think in a structured way.
- **Easy integration** – Feed the output into databases, UIs, or other APIs.

---

## 2. Why Zod?

[Zod](https://zod.dev/) is a TypeScript‑first schema declaration and validation library. It lets you define a schema for your expected data, then parse unknown input against that schema – throwing clear errors if something is wrong.

**Benefits**:
- **Type inference** – Zod automatically infers the TypeScript type from your schema.
- **Composable** – Build complex schemas from simple ones (objects, arrays, unions, etc.).
- **Runtime validation** – Catches malformed LLM responses before they crash your app.
- **Clear error messages** – Tells you exactly which field failed and why.

## 3. Generating Structured Outputs from LLMs

### 3.1 OpenAI JSON Mode

OpenAI’s Chat Completions API supports a `response_format` parameter that forces the model to return valid JSON.

```javascript
const response = await openai.chat.completions.create({
  model: 'gpt-4o-mini',
  messages: [
    { role: 'system', content: 'You are a helpful assistant. Always respond in JSON.' },
    { role: 'user', content: 'Extract name, age, and city from: "John is 30 and lives in Paris."' }
  ],
  response_format: { type: 'json_object' }
});

const raw = response.choices[0].message.content;
// raw is a JSON string like '{"name":"John","age":30,"city":"Paris"}'
```

**Limitations**:
- The model may still output invalid JSON or omit required fields.
- You still need to validate the shape of the JSON.

### 3.2 Function Calling / Tools

Function calling gives you even more control. You define a tool (function) with a strict JSON Schema, and the model returns a `tool_calls` object with arguments that exactly match that schema.

```javascript
const tools = [
  {
    type: 'function',
    function: {
      name: 'extract_person',
      description: 'Extract person details',
      parameters: {
        type: 'object',
        properties: {
          name: { type: 'string' },
          age: { type: 'number' },
          city: { type: 'string' }
        },
        required: ['name', 'age', 'city']
      }
    }
  }
];

const response = await openai.chat.completions.create({
  model: 'gpt-4o-mini',
  messages: [{ role: 'user', content: 'John is 30 and lives in Paris.' }],
  tools,
  tool_choice: { type: 'function', function: { name: 'extract_person' } }
});

const toolCall = response.choices[0].message.tool_calls[0];
const args = JSON.parse(toolCall.function.arguments); // { name: 'John', age: 30, city: 'Paris' }
```

Function calling is more reliable for structured data because the model is explicitly trained to fill the schema.

### 3.3 Other Providers

- **Anthropic Claude** – supports tool use (similar to OpenAI).
- **Google Gemini** – supports function calling.
- **Mistral / Cohere** – also have tool‑calling capabilities.

The concepts are identical; only the API shape differs.

## 4. Validating with Zod

Once you have a JSON object from the LLM, validate it with a Zod schema.

### 4.1 Define a Zod Schema

```typescript
import { z } from 'zod';

const PersonSchema = z.object({
  name: z.string(),
  age: z.number().int().positive(),
  city: z.string().min(1)
});

// Infer the TypeScript type
type Person = z.infer<typeof PersonSchema>; // { name: string; age: number; city: string }
```

### 4.2 Parse the LLM Output

```typescript
const raw = '{"name":"John","age":30,"city":"Paris"}'; // from LLM
try {
  const parsed = JSON.parse(raw);
  const person = PersonSchema.parse(parsed); // throws if invalid
  console.log(person); // { name: 'John', age: 30, city: 'Paris' }
} catch (e) {
  console.error('Validation failed:', e.errors);
  // handle error (e.g., retry, fallback)
}
```

### 4.3 Safe Parse

Use `safeParse` to get a result object without throwing:

```typescript
const result = PersonSchema.safeParse(parsed);
if (result.success) {
  // result.data is typed as Person
} else {
  // result.error contains validation issues
}
```

## 5. Complete Examples

### 5.1 Simple Example: Extract Person Information

```typescript
import OpenAI from 'openai';
import { z } from 'zod';

const openai = new OpenAI();

const PersonSchema = z.object({
  name: z.string(),
  age: z.number().int().positive(),
  city: z.string()
});

async function extractPerson(text: string) {
  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      { role: 'system', content: 'Extract person details as JSON.' },
      { role: 'user', content: text }
    ],
    response_format: { type: 'json_object' }
  });

  const raw = response.choices[0].message.content;
  if (!raw) throw new Error('No response');

  const parsed = JSON.parse(raw);
  return PersonSchema.parse(parsed);
}

// Usage
const person = await extractPerson('Alice is 25 and lives in London.');
console.log(person); // { name: 'Alice', age: 25, city: 'London' }
```


## 6. Error Handling and Retries

LLMs sometimes produce malformed JSON or omit required fields. Implement a retry mechanism with a corrected prompt.

```typescript
async function extractWithRetry<T>(
  schema: z.ZodSchema<T>,
  prompt: string,
  maxRetries = 3
): Promise<T> {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const response = await openai.chat.completions.create({
        model: 'gpt-4o-mini',
        messages: [
          { role: 'system', content: 'Return valid JSON matching the schema.' },
          { role: 'user', content: prompt }
        ],
        response_format: { type: 'json_object' }
      });

      const raw = response.choices[0].message.content!;
      const parsed = JSON.parse(raw);
      return schema.parse(parsed);
    } catch (e) {
      console.log(`Attempt ${i + 1} failed:`, e);
      if (i === maxRetries - 1) throw e;
      // Optionally add a note to the model about the error
      prompt += '\nPrevious attempt failed. Please ensure JSON is valid and includes all required fields.';
    }
  }
  throw new Error('Max retries exceeded');
}
```

## 7. Integration with LangChain

LangChain provides an `OutputParser` that can use a Zod schema via `StructuredOutputParser`.

```typescript
import { StructuredOutputParser } from 'langchain/output_parsers';
import { ChatPromptTemplate } from '@langchain/core/prompts';
import { ChatOpenAI } from '@langchain/openai';
import { z } from 'zod';

const schema = z.object({
  name: z.string(),
  age: z.number(),
  city: z.string()
});

const parser = StructuredOutputParser.fromZodSchema(schema);

const prompt = ChatPromptTemplate.fromTemplate(
  `Extract information from the following text:
{input}
{format_instructions}`
);

const chain = prompt.pipe(new ChatOpenAI({ model: 'gpt-4o-mini' })).pipe(parser);

const result = await chain.invoke({
  input: 'John is 30 and lives in Paris.',
  format_instructions: parser.getFormatInstructions()
});

console.log(result); // { name: 'John', age: 30, city: 'Paris' }
```

LangChain automatically injects format instructions and parses the output using your Zod schema.

## 8. Best Practices

- **Always validate** – Never trust LLM output directly.
- **Use function calling** for critical structured data – it's more reliable than JSON mode.
- **Keep schemas simple** – Complex nested structures increase chance of failure.
- **Provide examples** in the prompt (few‑shot) to guide the model.
- **Handle partial failures** – Use `partial()` or `.optional()` for non‑critical fields.
- **Log validation errors** to improve prompts over time.
- **Combine with guardrails** (moderation, PII redaction) for safety.

```typescript
const UserSchema = z.object({
  name: z.string(),
```

### 9.2 Refinements

Add custom validation logic:

```typescript
const AgeSchema = z.number().refine(age => age >= 0 && age <= 120, {
  message: 'Age must be between 0 and 120'
});
```

### 9.3 Default Values

```typescript

```

## 10. Common Pitfalls

- **Model returns string instead of JSON** – Use `response_format: { type: 'json_object' }` or function calling.
- **Missing required fields** – Prompt the model explicitly; use retries.
- **Wrong data types** (e.g., string instead of number) – Zod catches this; you can then refine the prompt.
- **Top‑level array in JSON mode** – OpenAI's JSON mode expects an object at the top level. If you need an array, wrap it: `{ "items": [...] }`.
- **Token limit exceeded** – Keep schemas and prompts concise.
