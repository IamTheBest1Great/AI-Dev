# 📘 AI Orchestration Frameworks — LangChain.js, LlamaIndex.ts & Selection Guide

This guide unifies two detailed sets of notes on building AI applications with orchestrators. It covers **LangChain.js** (general‑purpose chains, agents, LCEL) and **LlamaIndex.ts** (retrieval‑first indexing), from basic concepts to advanced retrieval strategies, with a framework selection decision matrix. Every concept and code snippet from both sources is preserved and logically combined.

---

## 1. Why Use an Orchestration Framework?

Building AI applications involves many moving parts:
- Calling LLMs (prompt engineering, multi‑turn history, streaming)
- Loading and splitting documents from various sources
- Generating embeddings
- Storing and querying vector databases
- Chaining multiple steps (retrieve → generate)
- Giving LLMs access to tools / function calling

Frameworks like **LangChain.js** and **LlamaIndex.ts** provide reusable abstractions and pre‑built components that glue these pieces together. They let you focus on **application logic** instead of reinventing infrastructure.

## 2. LangChain.js

LangChain.js is a comprehensive, modular framework for building LLM‑powered applications in TypeScript/JavaScript. It shines when you need **chains**, **agents**, and **complex workflows**.

### 2.1 Installation

```bash
npm install langchain @langchain/openai @langchain/pinecone @langchain/community
```

### 2.2 Core Concepts

| Concept | Description |
|---------|-------------|
| **Model** | LLM wrapper (e.g., `ChatOpenAI`, `OpenAIEmbeddings`) |
| **Prompt** | Template for sending instructions to the LLM |
| **Chain** | Sequence of calls (prompt → model → output parser) |
| **Document** | A piece of text with metadata |
| **Document Loader** | Loads data from sources (PDF, CSV, web, etc.) |
| **Text Splitter** | Splits documents into chunks |
| **Vector Store** | Wrapper around a vector database (Pinecone, Supabase pgvector, Memory) |
| **Retriever** | Interface for fetching relevant documents from a vector store |
| **Tool** | A callable function exposed to the LLM (calculator, search, API) |
| **Agent** | An LLM that decides which tools to use and in what order |

### 2.3 LCEL – LangChain Expression Language

LCEL is the declarative way to compose chains in LangChain.js. It uses the `|` pipe operator to connect components, much like Unix pipes. All key components implement the `Runnable` interface, enabling this composition.

```javascript
import { ChatPromptTemplate } from "@langchain/core/prompts";
import { ChatOpenAI } from "@langchain/openai";

const model = new ChatOpenAI({ model: "gpt-4o-mini" });
const prompt = ChatPromptTemplate.fromTemplate("Tell me a joke about {topic}");

const chain = prompt.pipe(model);
const response = await chain.invoke({ topic: "programmers" });
console.log(response.content);
```

LCEL handles streaming, batching, retries, and more automatically.

### 2.4 Simple LLM Call (without LCEL)

```typescript
import { ChatOpenAI } from '@langchain/openai';
import { HumanMessage } from '@langchain/core/messages';

const model = new ChatOpenAI({ model: 'gpt-4o-mini', temperature: 0 });
const response = await model.invoke([new HumanMessage('Tell me a joke.')]);
console.log(response.content);
```

### 2.5 Document Loaders, Splitters, and Retrievers

- **Document Loaders** – import raw data from 20+ formats (PDF, Word, HTML, CSV).  
- **Text Splitters** – break large documents into chunks optimized for embedding models and context windows.  
- **Retrievers** – the most common is `VectorStoreRetriever`, which fetches documents similar to a query.

### 2.6 Full RAG Pipeline Example (Pinecone + createRetrievalChain)

This example loads a PDF, splits it, stores embeddings in Pinecone, and builds a retrieval chain.

```typescript
import { PDFLoader } from 'langchain/document_loaders/fs/pdf';
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';
import { OpenAIEmbeddings } from '@langchain/openai';
import { PineconeStore } from '@langchain/pinecone';
import { Pinecone } from '@pinecone-database/pinecone';
import { ChatOpenAI } from '@langchain/openai';
import { createRetrievalChain } from 'langchain/chains/retrieval';
import { createStuffDocumentsChain } from 'langchain/chains/combine_documents';
import { ChatPromptTemplate } from '@langchain/core/prompts';

// 1. Load document
const loader = new PDFLoader('./document.pdf');
const docs = await loader.load();

// 2. Split into chunks
const splitter = new RecursiveCharacterTextSplitter({
  chunkSize: 1000,
  chunkOverlap: 200,
});
const splitDocs = await splitter.splitDocuments(docs);

// 3. Create embeddings and vector store (Pinecone)
const embeddings = new OpenAIEmbeddings({ model: 'text-embedding-3-small' });
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index(process.env.PINECONE_INDEX);
const vectorStore = await PineconeStore.fromDocuments(splitDocs, embeddings, { pineconeIndex: index });

// 4. Build retriever
const retriever = vectorStore.asRetriever({ k: 5 });

// 5. Create the chain
const llm = new ChatOpenAI({ model: 'gpt-4o-mini', temperature: 0 });
const prompt = ChatPromptTemplate.fromTemplate(`
Answer the question based only on the following context:
{context}

Question: {input}
`);
const documentChain = await createStuffDocumentsChain({ llm, prompt });
const retrievalChain = await createRetrievalChain({ retriever, combineDocsChain: documentChain });

// 6. Ask a question
const result = await retrievalChain.invoke({ input: 'What is the main topic?' });
console.log(result.answer);
```

### 2.7 Alternative RAG Pipeline (MemoryVectorStore + RunnableSequence)

Another common pattern uses a local `MemoryVectorStore` and `RunnableSequence` for a lightweight chain.

```javascript
import { RecursiveCharacterTextSplitter } from "langchain/text_splitter";
import { OpenAIEmbeddings } from "@langchain/openai";
import { MemoryVectorStore } from "langchain/vectorstores/memory";

const splitter = new RecursiveCharacterTextSplitter({ chunkSize: 1000 });
const docs = await splitter.splitDocuments(loadedDocs);

const vectorStore = await MemoryVectorStore.fromDocuments(
  docs,
  new OpenAIEmbeddings()
);
const retriever = vectorStore.asRetriever();

// Create a chain using RunnableSequence
const ragChain = RunnableSequence.from([
  {
    context: retriever.pipe(formatDocs),   // formatDocs is a helper to combine documents
    question: (input) => input.question,
  },
  prompt,
  model,
  outputParser,
]);

const result = await ragChain.invoke({ question: "Your question" });
```

### 2.8 Tools and Agents

LangChain supports agents that can decide which tools to use. For example, a simple ReAct agent with a search tool:

```typescript
import { createReactAgent } from '@langchain/langgraph/prebuilt';
import { TavilySearchResults } from '@langchain/community/tools/tavily_search';

const tools = [new TavilySearchResults({ maxResults: 1 })];
const agent = await createReactAgent({ llm, tools });
const result = await agent.invoke({ messages: [{ role: 'user', content: "What's the weather in Paris?" }] });
```

### 2.9 When to Use LangChain.js vs. Raw API

| Pros of LangChain | Cons of LangChain |
|-------------------|-------------------|
| **Faster prototyping** – pre‑built components and integrations | **Learning curve** – abstractions require time to understand |
| **Vendor agnostic** – easily swap between LLM providers or vector stores | **Complexity overhead** – overkill for very simple tasks |
| **Rich ecosystem** – connectors for databases, APIs, tools | **Debugging difficulty** – abstracted layers can hide issues |

**Rule of thumb**: Start with direct API calls for simple tasks. Adopt LangChain when your workflow becomes complex or you need to switch providers frequently.

## 3. LlamaIndex.ts

LlamaIndex.ts (formerly GPT Index) is a **retrieval‑first** framework. It focuses on indexing custom data and making it easily queryable. Its higher‑level abstractions dramatically reduce boilerplate for RAG.

### 3.1 Installation

```bash
npm install llamaindex @llamaindex/openai @llamaindex/pinecone
```

### 3.2 Core Concepts

| Concept | Description |
|---------|-------------|
| **Document** | A container for text and metadata |
| **Node** | A chunk of a document (like LangChain’s Document after splitting) |
| **Embedding** | Generated by an embedding model (e.g., OpenAI) |
| **VectorStore** | Interface to store nodes and their embeddings |
| **Index** | A data structure that organises nodes for fast retrieval |
| **QueryEngine** | The main interface for asking questions over an index – encapsulates retrieval + synthesis |
| **Retriever** | Fetches nodes based on a query |
| **ResponseSynthesizer** | Takes retrieved nodes and produces a final answer using an LLM |

### 3.3 Data Ingestion and Indexing (Three‑Layer Architecture)

1. **Data Connectors** – Ingest from 160+ sources (PDFs, APIs, databases, etc.).  
2. **Parsing / Chunking** – Built‑in strategies like fixed‑size, sentence‑based, or semantic chunking.  
3. **Indexes** – Data structures optimised for retrieval (most common: `VectorStoreIndex`).

### 3.4 RAG Pipeline with LlamaIndex.ts (Pinecone)

```typescript
import {
  Document,
  VectorStoreIndex,
  SimpleDirectoryReader,
  OpenAIEmbedding,
  OpenAI,
  Settings,
} from 'llamaindex';
import { PineconeVectorStore } from '@llamaindex/pinecone';
import { Pinecone } from '@pinecone-database/pinecone';

// Optional: configure global defaults
Settings.llm = new OpenAI({ model: 'gpt-4o-mini', temperature: 0 });
Settings.embedModel = new OpenAIEmbedding({ model: 'text-embedding-3-small' });

// 1. Load document (directly or from a directory)
const document = new Document({ text: '...', metadata: { source: 'file.pdf' } });

// 2. Set up Pinecone vector store
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const pineconeIndex = pc.index(process.env.PINECONE_INDEX);
const vectorStore = new PineconeVectorStore({ pineconeIndex });

// 3. Create index – automatically embeds and stores nodes
const index = await VectorStoreIndex.fromDocuments([document], { vectorStore });

// 4. Create a query engine
const queryEngine = index.asQueryEngine({
  retriever: { similarityTopK: 5 },   // retrieve top 5 chunks
});

// 5. Ask a question
const response = await queryEngine.query({ query: 'What is the main topic?' });
console.log(response.toString());
```

LlamaIndex handles chunking, embedding, and retrieval with sensible defaults.

### 3.5 Custom Chunking

You can customise the node parser:

```typescript
import { SentenceSplitter } from 'llamaindex';

const splitter = new SentenceSplitter({ chunkSize: 1024, chunkOverlap: 200 });
const nodes = splitter.splitDocuments([document]);
```

Then build an index directly from the nodes using `VectorStoreIndex.fromDocuments`.

### 3.6 Advanced Retrieval Strategies

LlamaIndex supports several strategies beyond simple top‑K:

- **Re‑ranking** – Use a cross‑encoder model to refine the top‑retrieved results after the initial vector search, improving relevance.
- **MMR (Maximum Marginal Relevance)** – Balances **relevance** and **diversity**, preventing near‑duplicate results in your answer set.
- **Hybrid Search (BM25 + Vector)** – Combine keyword search (BM25) with vector similarity for robust retrieval, especially when exact terms matter (product codes, names). This is often implemented via `HybridRetriever` that merges results from a keyword retriever and a vector retriever.

*Why these matter*: Plain vector search can miss rare keywords or over‑focus on one aspect. These strategies improve recall and diversity.

## 4. LangChain vs. LlamaIndex: Feature Comparison

| Aspect | LangChain.js | LlamaIndex.ts |
|--------|-------------|---------------|
| **Philosophy** | General‑purpose AI orchestrator (chains, agents, tools, memory) | Retrieval‑first, centred on data indexing and querying |
| **Abstraction level** | Lower; you compose components explicitly | Higher; many defaults, minimal code for RAG |
| **Flexibility** | Extremely flexible – build any LLM workflow | Flexible but best for retrieval‑augmented tasks |
| **Learning curve** | Steeper; more concepts to learn | Gentler for pure RAG use cases |
| **Agents & tools** | Excellent built‑in support (ReAct, OpenAI tools, etc.) | Agents exist but are less mature than LangChain’s |
| **Integration ecosystem** | Very large (hundreds of connectors) | Good and growing, especially for vector stores |
| **Best for** | Complex chains, stateful agents, multi‑step reasoning | Quick, robust RAG over custom data; heavy document processing |

## 5. Framework Selection Guide

Use this matrix to choose the right tool for your project.

| Use Case | Recommended Approach |
|----------|----------------------|
| **Simple completions, full control over prompt logic** | **Raw API** – No abstraction overhead; every call is explicit. |
| **Rapid prototyping, standard RAG, vendor‑switching** | **LangChain.js** – Rich integrations and pre‑built components. |
| **Heavy document processing, complex retrieval (re‑ranking, hybrid)** | **LlamaIndex.ts** – Built‑in advanced retrieval strategies, data connectors, and indexing. |
| **Stateful, multi‑step agent workflows (cycles, human‑in‑the‑loop)** | **LangGraph** (on top of LangChain) – Designed for cyclical, conditional, and multi‑agent logic. |

### Key decision criteria

- **Control vs. Speed**: Raw API gives maximum control; frameworks accelerate development.
- **RAG Complexity**: If your code revolves around data, use LlamaIndex; if it’s part of a larger workflow, LangChain may be more flexible.
- **Agentic Logic**: For multi‑turn autonomous agents with tool calling, LangChain Agents or LangGraph are the best choices.

## 6. Best Practices (Applies to Both Frameworks)

- **Set `temperature=0`** for factual Q&A to reduce hallucinations.
- **Experiment with chunk sizes** – too small loses context, too large dilutes relevance.
- **Use metadata filtering** to limit retrieval scope when the corpus is large (e.g., by source, date, category).
- **Cache embeddings** for frequently recurring queries – saves time and money.
- **Monitor costs** – each API call costs money; summaries and log usage.
- **Keep up with updates** – both libraries evolve fast; always check the latest documentation.
