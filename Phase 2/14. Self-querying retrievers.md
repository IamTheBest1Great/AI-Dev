# 📘 Self‑Querying Retrievers — Complete Guide

A **self‑querying retriever** is an advanced RAG component that uses an LLM to transform a user’s natural language query into a structured search request. It not only generates a search term but also extracts **metadata filters** (like dates, categories, or tags) from the query, then applies them during retrieval.

This guide explains what self‑querying retrievers are, why they matter, and how to implement them in your RAG pipeline using LangChain.js.

---

## 1. What Is a Self‑Querying Retriever?

Traditional vector search retrieves documents solely based on semantic similarity. If a user asks for *“movies by Christopher Nolan after 2010”*, a pure vector search might return movies about Nolan or movies from 2010, but it won’t reliably **combine** the director filter and the year filter.

A self‑querying retriever solves this by:

1. **Parsing** the user’s question into:
   - A **query string** (for semantic search)
   - **Metadata filters** (structured conditions like `director = "Christopher Nolan"` and `year >= 2010`)

2. **Executing** the retrieval with both the query embedding and the filters, so results satisfy both the semantic meaning and the explicit constraints.

It’s called “self‑querying” because the retriever itself (with the help of an LLM) figures out how to query the underlying vector store.

---

## 2. Why Use Self‑Querying Retrievers?

- **Precision** – Enables users to ask complex, filtered questions without you hard‑coding filters.
- **User experience** – Natural language becomes a powerful query language.
- **Automation** – No need to build separate UI elements for filtering; the LLM extracts filters automatically.
- **Efficiency** – Reduces the number of irrelevant documents by applying metadata filters early.

---

## 3. How It Works

```
┌─────────────────┐
│  User Query     │
│ "movies by Nolan│
│  after 2010"    │
└────────┬────────┘
         ▼
┌─────────────────┐
│  LLM (with      │   <-- Uses a prompt to convert the query
│  QueryConstructor│       into a structured object
└────────┬────────┘
         │  Output: { query: "movies", filter: { director: "Nolan", year: { $gte: 2010 } } }
         ▼
┌─────────────────┐
│  Vector Store   │   <-- Retrieves documents that match the semantic query AND the filter
└────────┬────────┘
         ▼
┌─────────────────┐
│  Retrieved Docs │
└─────────────────┘
```

The core components are:

- **Metadata** – Each document in your vector store must have metadata fields (e.g., `director`, `year`, `category`).
- **Query Constructor** – An LLM chain that takes the user query and outputs a structured query object.
- **Retriever** – The vector store retriever that accepts both a query string and metadata filters.

## 4. Implementing Self‑Querying Retrievers with LangChain

LangChain provides a ready‑to‑use `SelfQueryRetriever` that handles the LLM parsing and filter application. You need:

1. A **vector store** that supports metadata filtering (e.g., Pinecone, Supabase pgvector, Chroma).
2. A **document description** (so the LLM knows what metadata fields exist).
3. An **LLM** (ChatOpenAI, etc.) to do the parsing.

### 4.1 Setting Up Metadata and Document Descriptions

First, define the metadata fields your documents have. This description is passed to the LLM so it knows what filters it can apply.

```typescript
import { AttributeInfo } from 'langchain/schema/query_constructor';

const attributes: AttributeInfo[] = [
  {
    name: 'director',
    description: 'Director of the movie',
    type: 'string',
  },
  {
    name: 'year',
    description: 'Year the movie was released',
    type: 'number',
  },
  {
    name: 'genre',
    description: 'Genre of the movie',
    type: 'string',
  },
];
```

### 4.2 Create the Vector Store with Metadata

Assume you have documents (movies) stored in a vector store. When adding documents, include metadata.

```typescript
import { OpenAIEmbeddings } from '@langchain/openai';
import { PineconeStore } from '@langchain/pinecone';
import { Pinecone } from '@pinecone-database/pinecone';

const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index('movies-index');

const vectorStore = await PineconeStore.fromExistingIndex(
  new OpenAIEmbeddings(),
  { pineconeIndex: index }
);
```

Make sure your index stores metadata (Pinecone does by default; for others, you may need to configure).

### 4.3 Create the Self‑Query Retriever

```typescript
import { SelfQueryRetriever } from 'langchain/retrievers/self_query';
import { OpenAI } from '@langchain/openai';
import { PineconeTranslator } from 'langchain/retrievers/self_query/pinecone';

const llm = new OpenAI({ modelName: 'gpt-4o-mini', temperature: 0 });

const retriever = await SelfQueryRetriever.fromLLM({
  llm,
  vectorStore,
  documentContents: 'Brief summary of a movie', // description of the document content
  attributeInfo: attributes,
  structuredQueryTranslator: new PineconeTranslator(), // translator for your vector store
  searchParams: {
    k: 5, // number of documents to retrieve
  },
});
```

**Explanation**:
- `documentContents` tells the LLM what kind of text is stored (helps with query extraction).
- `attributeInfo` lists available metadata fields.
- `structuredQueryTranslator` converts the structured query into the filter syntax of your vector store (e.g., Pinecone, Supabase, Chroma). LangChain provides translators for each supported vector store.
- `searchParams` includes the default `k` (number of results).

### 4.4 Using the Retriever

Now you can use the retriever just like any other LangChain retriever:

```typescript
const query = 'movies by Christopher Nolan after 2010';
const docs = await retriever.getRelevantDocuments(query);
console.log(docs);
```

Behind the scenes:
1. The LLM parses the query into something like: `{ query: 'movies', filter: { director: 'Christopher Nolan', year: { $gte: 2010 } } }`
2. The translator converts the filter to Pinecone syntax: `{ director: { $eq: 'Christopher Nolan' }, year: { $gte: 2010 } }`
3. The vector store performs a similarity search on `'movies'` with those filters applied.
4. Only movies that match both the semantic meaning and the filters are returned.

### 4.5 Customizing the Prompt

If the default prompt isn't working well, you can create a custom prompt. The prompt is responsible for generating a structured query.

```typescript
import { PromptTemplate } from '@langchain/core/prompts';

const customPrompt = new PromptTemplate({
  template: `
Your task is to convert a user question into a structured query.
You have access to the following metadata attributes:
{attributes}

User question: {question}

Return a JSON object with "query" (the search text) and "filter" (a filter object).
If no filter is needed, set filter to null.
`,
  inputVariables: ['attributes', 'question'],
});

const retriever = await SelfQueryRetriever.fromLLM({
  llm,
  vectorStore,
  documentContents: 'Movie summary',
  attributeInfo: attributes,
  structuredQueryTranslator: new PineconeTranslator(),
  prompt: customPrompt,
});
```

## 5. Complete Example: Movie Database

Let's build a complete self‑querying retriever for a movie collection.

### 5.1 Add Documents with Metadata

```typescript
// Assume you have movie objects
const movies = [
  {
    id: '1',
    title: 'Inception',
    director: 'Christopher Nolan',
    year: 2010,
    genre: 'Sci-Fi',
    plot: 'A thief who steals corporate secrets...',
  },
  // ... more
];

// Add to vector store with metadata
await vectorStore.addDocuments(
  movies.map(m => ({
    pageContent: m.plot,
    metadata: {
      title: m.title,
      director: m.director,
      year: m.year,
      genre: m.genre,
    },
  }))
);
```

### 5.2 Define Attributes and Create Retriever

```typescript
const attributes = [
  { name: 'director', description: 'Director of the movie', type: 'string' },
  { name: 'year', description: 'Release year', type: 'number' },
  { name: 'genre', description: 'Movie genre', type: 'string' },
];

const llm = new OpenAI({ modelName: 'gpt-4o-mini', temperature: 0 });

const retriever = await SelfQueryRetriever.fromLLM({
  llm,
  vectorStore,
  documentContents: 'Movie plot summary',
  attributeInfo: attributes,
  structuredQueryTranslator: new PineconeTranslator(),
});
```

### 5.3 Query

```typescript
const docs = await retriever.getRelevantDocuments(
  'sci-fi movies directed by Nolan after 2010'
);
// This should return Inception (matches genre, director, year) and maybe others.
```

## 6. Supported Filter Operators

The self‑query retriever supports a variety of operators, depending on the translator:

- **Comparison**: `$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`
- **Logical**: `$and`, `$or` (composed automatically)
- **List operations**: `$in`, `$nin` (for array fields)

The LLM will automatically construct the appropriate filter object based on the query.

## 7. Combining with Other Techniques

### 7.1 Self‑Query + Multi‑Query

You can wrap a self‑query retriever inside a multi‑query retriever to first generate multiple queries, then parse each with the self‑query logic. LangChain doesn't provide this out of the box, but you can implement a custom retriever that does both.

### 7.2 Self‑Query + Hybrid Search

If your vector store supports hybrid search (vector + keyword), you can combine them. The self‑query retriever only handles the metadata filter; the underlying similarity search can be hybrid if configured in the vector store.

### 7.3 Self‑Query + Re‑ranking

After retrieving with filters, you can pass the results through a re‑ranker for even better ordering.

---

## 8. Best Practices

- **Provide clear attribute descriptions** – The LLM needs to know what each metadata field means to use it correctly.
- **Keep metadata values clean** – Inconsistent values (e.g., "Nolan" vs "Christopher Nolan") will break filtering.
- **Test with diverse queries** – Ensure the LLM correctly extracts filters.
- **Handle missing filters gracefully** – If the query doesn't specify a filter, the retriever should just do semantic search.
- **Monitor LLM costs** – Each query uses an LLM call; cache if possible.
- **Use a translator that matches your vector store** – Pinecone, Chroma, Supabase, etc., all have different filter syntaxes.

---

## 9. Limitations and Considerations

- **LLM may misinterpret** – Complex or ambiguous queries can lead to incorrect filters.
- **Filter field names must match exactly** – The LLM uses the attribute names you provide.
- **Not all vector stores support the same filter operators** – Ensure your translator matches your store's capabilities.
- **Latency** – Adds an extra LLM call before retrieval.
