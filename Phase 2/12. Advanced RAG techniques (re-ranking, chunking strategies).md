# 📘 Advanced RAG Techniques: Re-ranking & Chunking Strategies — Complete Guide

You've mastered basic RAG: load documents, chunk, embed, store, retrieve, and generate. But as you scale to complex documents and real‑world queries, you'll encounter challenges:

- **Chunking** – How big should chunks be? How do you avoid cutting off context?
- **Retrieval quality** – The top‑k vectors may not always be the most relevant.
- **Noise** – Irrelevant chunks can dilute the LLM's answer.

**Advanced RAG** techniques address these issues. In this guide, we dive deep into two critical improvements: **chunking strategies** and **re-ranking**. You'll learn how to tune them for your specific use case and see code examples you can apply immediately.

---

## 1. Chunking Strategies: The Foundation of Good Retrieval

Chunking is the process of splitting documents into smaller pieces before embedding. The way you chunk has a massive impact on retrieval accuracy.

### 1.1 Why Chunking Matters

- **Smaller chunks** increase precision (each chunk focuses on one topic), but may lose surrounding context.
- **Larger chunks** preserve context but may introduce noise and exceed the LLM's context window.
- **Overlap** helps maintain continuity at boundaries.

### 1.2 Common Chunking Methods

#### A. Fixed‑Size Chunking (with Overlap)
The simplest approach: split by a fixed number of characters or tokens, with an optional overlap.

```javascript
function fixedSizeChunk(text, chunkSize = 1000, overlap = 200) {
  const chunks = [];
  let start = 0;
  while (start < text.length) {
    const end = Math.min(start + chunkSize, text.length);
    chunks.push(text.substring(start, end));
    start += chunkSize - overlap;
  }
  return chunks;
}
```

**Pros:** Simple, fast.  
**Cons:** May cut sentences or paragraphs in half, losing meaning.

#### B. Recursive Character Text Splitting
LangChain's `RecursiveCharacterTextSplitter` tries to keep paragraphs, sentences, then words together. It recursively splits on different separators.

```javascript
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';

const splitter = new RecursiveCharacterTextSplitter({
  chunkSize: 1000,
  chunkOverlap: 200,
  separators: ['\n\n', '\n', '.', ' ', '']  // default order
});
const docs = await splitter.createDocuments([text]);
```

**Pros:** Respects document structure, produces semantically coherent chunks.  
**Cons:** Still may split sentences if needed.

#### C. Semantic Chunking (by Topic / Embedding Similarity)
Idea: merge sentences until the embedding similarity between successive sentences drops below a threshold, indicating a topic shift.

```javascript
async function semanticChunking(sentences, threshold = 0.7) {
  const chunks = [];
  let currentChunk = [sentences[0]];
  let currentEmbedding = await embed(sentences[0]);
  
  for (let i = 1; i < sentences.length; i++) {
    const nextEmbedding = await embed(sentences[i]);
    const similarity = cosineSimilarity(currentEmbedding, nextEmbedding);
    if (similarity > threshold) {
      currentChunk.push(sentences[i]);
    } else {
      chunks.push(currentChunk.join(' '));
      currentChunk = [sentences[i]];
      currentEmbedding = nextEmbedding;
    }
  }
  chunks.push(currentChunk.join(' '));
  return chunks;
}
```

**Pros:** Creates topically coherent chunks, excellent for long documents.  
**Cons:** Requires many embedding calls (can be batched); slower.

#### D. Agentic / LLM‑based Chunking
Use an LLM to decide where to split. For example, ask the LLM to identify natural breakpoints (section boundaries, topic shifts). This is overkill for most applications but can be useful for complex documents.

### 1.3 Best Practices

- **Chunk size**: 500–1500 tokens is a good starting point. Test with your data.
- **Overlap**: 10–20% of chunk size helps maintain context across boundaries.
- **Document type**:
  - **Articles/books**: Use recursive splitter with `\n\n` as primary separator.
  - **Code**: Use language‑specific splitters (e.g., `RecursiveCharacterTextSplitter` with Python separators).
  - **Conversations**: Split by speaker turn.
- **Metadata**: Always store source, page number, chunk index in metadata for traceability.

### 1.4 Measuring Chunking Quality

You can evaluate chunking by:
- **Retrieval precision** on a test set of questions.
- **Answer completeness** – does the top chunk contain enough info to answer?
- **Context coherence** – are ideas cut off mid‑sentence?

## 2. Re-ranking: Picking the Best Chunks

Even with optimal chunking, the top‑k vectors from similarity search are not guaranteed to be the most **useful** for answering a question. Re-ranking adds a second step: after retrieving candidate chunks, a more powerful (but slower) model re‑scores them based on deeper relevance.

### 2.1 Why Re‑rank?

- **Bi‑encoders** (used in vector search) embed query and documents separately; they are fast but lose interaction between query and document.
- **Cross‑encoders** take the query and document together as input, allowing full attention. They are much more accurate, but too slow to run on the entire corpus.

**Re‑ranking workflow:**
1. Retrieve `k` candidates (e.g., 20–100) using fast vector search.
2. Pass each candidate + query to a cross‑encoder to get a relevance score.
3. Re‑order candidates by the new score, take top‑n (e.g., 5) for the LLM.

### 2.2 Re‑ranking Models

Popular cross‑encoders (available via Hugging Face, Cohere, etc.):

- **Cohere Rerank** – commercial API, easy to use, supports multiple languages.
- **BAAI/bge-reranker-v2-m3** – open‑source, very strong.
- **cross‑encoder/ms-marco-MiniLM-L-6-v2** – lightweight, good for English.

### 2.3 Implementing Re‑ranking

#### A. Using Cohere Rerank API (JavaScript)

```javascript
import { CohereClient } from 'cohere-ai';

const cohere = new CohereClient({ token: process.env.COHERE_API_KEY });

async function rerankWithCohere(query, documents, topN = 5) {
  const response = await cohere.rerank({
    model: 'rerank-english-v3.0', // or 'rerank-multilingual-v3.0'
    query: query,
    documents: documents.map(doc => doc.text), // array of strings
    topN: topN,
  });
  
  // response.results contains indices and relevance scores
  return response.results.map(r => ({
    ...documents[r.index],
    relevanceScore: r.relevance_score,
  }));
}
```

#### B. Using a Local Cross‑Encoder with Transformers.js (Node.js)

```javascript
import { pipeline } from '@xenova/transformers';

// Load cross‑encoder model
const reranker = await pipeline('text-classification', 'Xenova/ms-marco-MiniLM-L-6-v2');

async function rerankLocal(query, documents, topN = 5) {
  const pairs = documents.map(doc => ({ text: query, text_pair: doc.text }));
  const scores = await Promise.all(pairs.map(p => reranker(p)));
  const scoredDocs = documents.map((doc, i) => ({
    ...doc,
    relevanceScore: scores[i].score,
  }));
  scoredDocs.sort((a, b) => b.relevanceScore - a.relevanceScore);
  return scoredDocs.slice(0, topN);
}
```

**Note:** Running cross‑encoders locally requires Node.js with good CPU/GPU; for production, consider a dedicated inference service.

### 2.4 Integrating Re‑ranking into a RAG Pipeline

```javascript
async function advancedRAG(query) {
  // 1. Retrieve many candidates (e.g., 20) via vector search
  const candidates = await vectorStore.similaritySearch(query, 20);
  
  // 2. Re‑rank using Cohere
  const reranked = await rerankWithCohere(query, candidates, 5);
  
  // 3. Generate answer with top‑5 chunks
  const context = reranked.map(r => r.text).join('\n\n');
  const answer = await generateAnswer(query, context);
  
  return { answer, sources: reranked };
}
```

### 2.5 Evaluation of Re‑ranking

Measure:
- **NDCG@k**, **MAP@k** – how well the new order matches ideal relevance.
- **Answer accuracy** – does re‑ranking improve the final answer?
- **Latency** – re‑ranking adds overhead; you may need to cache results.

## 3. Combining Chunking and Re‑ranking: A Complete Advanced RAG Example

Let's put it all together with a realistic scenario: indexing a long technical document.

```javascript
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';
import { OpenAIEmbeddings } from '@langchain/openai';
import { PineconeStore } from '@langchain/pinecone';
import { Pinecone } from '@pinecone-database/pinecone';
import { CohereClient } from 'cohere-ai';
import OpenAI from 'openai';

// Initialize clients
const openai = new OpenAI();
const cohere = new CohereClient({ token: process.env.COHERE_API_KEY });
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index('my-docs');

// 1. Advanced chunking (recursive with metadata)
async function chunkDocument(text, metadata) {
  const splitter = new RecursiveCharacterTextSplitter({
    chunkSize: 1000,
    chunkOverlap: 200,
  });
  const docs = await splitter.createDocuments([text]);
  return docs.map((doc, i) => ({
    ...doc,
    metadata: { ...metadata, chunkIndex: i }
  }));
}

// 2. Store with embeddings
async function indexDocument(text, metadata) {
  const chunks = await chunkDocument(text, metadata);
  const embeddings = new OpenAIEmbeddings();
  await PineconeStore.fromDocuments(chunks, embeddings, { pineconeIndex: index });
}

// 3. Advanced retrieval with re‑ranking
async function retrieveAndRerank(query, topK = 20, finalK = 5) {
  // Get initial candidates from vector store
  const vectorStore = await PineconeStore.fromExistingIndex(
    new OpenAIEmbeddings(),
    { pineconeIndex: index }
  );
  const candidates = await vectorStore.similaritySearch(query, topK);
  
  // Re‑rank
  const documents = candidates.map(c => c.pageContent);
  const rerankResults = await cohere.rerank({
    model: 'rerank-english-v3.0',
    query,
    documents,
    topN: finalK,
  });
  
  // Map back to original documents with scores
  return rerankResults.results.map(r => ({
    ...candidates[r.index],
    relevanceScore: r.relevance_score,
  }));
}

// 4. Generate answer
async function ask(query) {
  const relevantChunks = await retrieveAndRerank(query);
  const context = relevantChunks.map(c => c.pageContent).join('\n\n');
  
  const completion = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      { role: 'system', content: 'Answer based on the context.' },
      { role: 'user', content: `Context:\n${context}\n\nQuestion: ${query}` }
    ],
  });
  return {
    answer: completion.choices[0].message.content,
    sources: relevantChunks.map(c => ({
      text: c.pageContent.slice(0, 200) + '...',
      score: c.relevanceScore,
    })),
  };
}
```

## 4. Advanced Chunking Techniques (Beyond Basics)

### 4.1 Hierarchical Chunking

Store documents in a hierarchy: document → sections → paragraphs → sentences. When retrieving, you can return the appropriate level based on the query.

**Use case:** Legal documents where you need to reference clauses within sections.

### 4.2 Metadata‑Aware Chunking

Attach rich metadata to chunks: title, headings, page numbers, timestamps. Then use filters to narrow search before vector similarity.

```javascript
// Store with metadata like headings
await vectorStore.addDocuments(chunks.map(chunk => ({
  pageContent: chunk.text,
  metadata: {
    heading: chunk.heading,
    page: chunk.page,
    docId: doc.id,
  }
})));
```

### 4.3 Indexing Multiple Representations

Store both the raw chunk and a summary of the chunk. At query time, search over summaries, then fetch the corresponding full chunk. This helps for very large chunks where summaries capture the gist.

### 4.4 Small‑to‑Big Retrieval

Retrieve small chunks (e.g., sentences) for high precision, but then return the surrounding larger block (e.g., paragraph or section) as context to the LLM. This gives the best of both worlds: precise retrieval and rich context.

```javascript
// Store small chunks with reference to parent larger chunk
const smallChunks = splitter.split(text, { chunkSize: 200 });
const parentMap = ... // mapping from small chunk to parent ID

// Retrieve small chunks, then fetch parent content
async function retrieveWithContext(query) {
  const smallResults = await vectorStore.similaritySearch(query, 5);
  const parentIds = [...new Set(smallResults.map(r => r.metadata.parentId))];
  const parentChunks = await fetchParents(parentIds);
  return parentChunks;
}
```

## 5. Best Practices Summary

- **Experiment with chunk sizes** – there's no one‑size‑fits‑all. Use an evaluation set.
- **Use overlap** to prevent cut‑off.
- **Store metadata** (source, headings, index) for traceability and filtering.
- **Re‑rank** when retrieval quality matters and you can afford the latency.
- **Cache re‑rank results** for repeated queries.
- **Consider hybrid search** (vector + keyword) before re‑ranking to improve recall.
