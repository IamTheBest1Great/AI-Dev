# 📘 Storing and Querying Vectors — Complete Guide

You've learned what embeddings are, how to generate them via API, and the role of vector databases. Now it's time to **actually store those vectors and query them** to build semantic search, RAG, and other AI applications.

This guide covers the core operations—**inserting vectors** and **performing similarity searches**—using two popular platforms: **Pinecone** (managed vector DB) and **Supabase pgvector** (PostgreSQL extension). By the end, you'll be able to:

- Store vectors with metadata in both systems
- Query for the most similar vectors to a user's query
- Apply metadata filters to narrow results
- Understand indexing for performance

---

## 1. Prerequisites

Before storing and querying, make sure you have:

- **Embeddings** generated (e.g., from OpenAI's `text-embedding-3-small`). We'll assume 1536‑dimensional vectors.
- **Access to a vector database**:
  - **Pinecone**: Account, API key, and an index created (see previous guide).
  - **Supabase**: Project, database URL, and the `vector` extension enabled.

- **Environment variables** set:
  ```
  OPENAI_API_KEY=sk-...
  PINECONE_API_KEY=...
  PINECONE_INDEX=my-index
  SUPABASE_URL=https://xyz.supabase.co
  SUPABASE_KEY=eyJ...
  ```

- **Install required packages**:
  ```bash
  npm install @pinecone-database/pinecone @supabase/supabase-js openai
  ```

## 2. Storing Vectors

Storing vectors means inserting records that contain:
- **ID** – A unique identifier (string)
- **Vector values** – An array of numbers (float)
- **Metadata** – Optional key‑value pairs for filtering

### 2.1 Data Model

Each record represents a chunk of text, an image, or any item you want to search semantically.

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique identifier (e.g., UUID or document ID + chunk index) |
| `values` | number[] | The embedding vector |
| `metadata` | object | Any additional fields (title, category, timestamp, etc.) |

### 2.2 Batching and Chunking

- **Batch size**: Insert vectors in batches of 100–1000 for efficiency. Most APIs have a per‑request limit (e.g., Pinecone accepts up to 1000 vectors per upsert).
- **Chunking**: If your vectors come from long documents, you already chunked them during embedding. Store each chunk as a separate vector record.

### 2.3 Storing in Pinecone

#### Initialize Pinecone client
```javascript
import { Pinecone } from '@pinecone-database/pinecone';

const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index(process.env.PINECONE_INDEX); // reference to your index
```

#### Upsert (insert or update) vectors
```javascript
async function storeVectors(vectors) {
  // vectors: array of { id, values, metadata }
  try {
    const result = await index.upsert(vectors);
    console.log(`Upserted ${vectors.length} vectors`);
    return result;
  } catch (error) {
    console.error('Upsert failed:', error);
    throw error;
  }
}

// Example usage
const myVectors = [
  {
    id: 'doc1-chunk1',
    values: [0.023, -0.045, ...], // 1536 numbers
    metadata: { title: 'AI Intro', chunk: 1, source: 'book.pdf' }
  },
  // ... more
];

await storeVectors(myVectors);
```

**Note**: Pinecone also supports **namespaces** to isolate data. If you want to use a namespace, specify it in the index reference:
```javascript
const index = pc.index(process.env.PINECONE_INDEX).namespace('my-namespace');
```

### 2.4 Storing in Supabase pgvector

#### Setup table with vector column
First, ensure you have a table with a `vector` column. Example SQL:
```sql
create table document_chunks (
  id uuid primary key default gen_random_uuid(),
  content text,
  metadata jsonb,
  embedding vector(1536)   -- match your embedding dimension
);
```

#### Insert vectors using Supabase JS
```javascript
import { createClient } from '@supabase/supabase-js';

const supabase = createClient(process.env.SUPABASE_URL, process.env.SUPABASE_KEY);

async function storeVectors(vectors) {
  // vectors: array of { content, metadata, embedding }
  const { data, error } = await supabase
    .from('document_chunks')
    .insert(vectors.map(v => ({
      content: v.content,
      metadata: v.metadata,
      embedding: v.embedding  // direct array of numbers
    })));

  if (error) throw error;
  return data;
}

// Example
const myVectors = [
  {
    content: 'The Eiffel Tower was completed in 1889...',
    metadata: { title: 'Eiffel Tower', category: 'history' },
    embedding: [0.023, -0.045, ...]  // from embedding API
  }
];

await storeVectors(myVectors);
```

**Important**: PostgreSQL expects the vector as a string like `'[0.023, -0.045, ...]'`? Actually, the `vector` type accepts an array literal in the form `'[0.023, -0.045, ...]'` but the Supabase JS client will automatically convert a JavaScript array to the correct format if the column type is `vector`. It does this via the PostgREST API, which expects a string representation. It's usually fine to pass an array; the client serializes it.

Alternatively, you can use the `rpc` method with a custom function to insert (see below for query function).

### 2.5 Handling Large Ingestion

For large datasets (millions of vectors), consider:
- **Parallel uploads**: Use `Promise.all` with concurrency control (e.g., limit to 10 concurrent batches).
- **Error handling**: Implement retries with exponential backoff.
- **Idempotency**: If you retry, ensure duplicates are handled (Pinecone upsert is idempotent by ID; in pgvector, use `on conflict`).

## 3. Querying Vectors

The goal: given a user's query, convert it to an embedding, then find the most similar vectors in your database.

### 3.1 Query Workflow

1. **Embed the query** using the same model used for stored vectors.
2. **Send the query vector** to the vector DB with a `topK` parameter (how many results to return).
3. **Optionally apply metadata filters** to restrict the search.
4. **Receive results** with similarity scores and metadata.

### 3.2 Querying in Pinecone

#### Basic similarity search
```javascript
async function querySimilar(queryText, topK = 10) {
  // 1. Generate embedding for query
  const embedding = await getEmbedding(queryText); // from OpenAI or other

  // 2. Query Pinecone
  const results = await index.query({
    vector: embedding,
    topK,
    includeMetadata: true,  // return metadata
    includeValues: false    // usually you don't need the raw vector
  });

  return results.matches; // array of { id, score, metadata }
}

// Usage
const matches = await querySimilar('famous landmarks in Paris');
matches.forEach(m => {
  console.log(`Score: ${m.score}, Content: ${m.metadata.content}`);
});
```

#### With metadata filtering
```javascript
const results = await index.query({
  vector: embedding,
  topK: 10,
  filter: {
    category: { "$eq": "history" },
    year: { "$gte": 1800 }
  },
  includeMetadata: true
});
```

Pinecone supports a rich filter language (`$eq`, `$ne`, `$gt`, `$gte`, `$lt`, `$lte`, `$in`, `$nin`, `$exists`, `$and`, `$or`).

### 3.3 Querying in Supabase pgvector

You have two options: use a SQL function (recommended) or write raw queries with the distance operator.

#### Create a match function (once)
```sql
create or replace function match_documents (
  query_embedding vector(1536),
  match_count int default 10,
  filter_category text default null
)
returns table (
  id uuid,
  content text,
  metadata jsonb,
  similarity float
)
language plpgsql
as $$
begin
  return query
  select
    document_chunks.id,
    document_chunks.content,
    document_chunks.metadata,
    1 - (document_chunks.embedding <=> query_embedding) as similarity
  from document_chunks
  where 
    (filter_category is null or document_chunks.metadata->>'category' = filter_category)
  order by document_chunks.embedding <=> query_embedding
  limit match_count;
end;
$$;
```

#### Query using the function
```javascript
async function querySimilar(queryText, topK = 10, category = null) {
  // 1. Embed query
  const embedding = await getEmbedding(queryText);

  // 2. Call the match function via RPC
  const { data, error } = await supabase
    .rpc('match_documents', {
      query_embedding: embedding,
      match_count: topK,
      filter_category: category
    });

  if (error) throw error;
  return data; // array of { id, content, metadata, similarity }
}
```

#### Direct SQL with operator (if not using function)
You can also use the `<=>` operator directly, but you need to cast the vector:
```javascript
const { data, error } = await supabase
  .from('document_chunks')
  .select('id, content, metadata, similarity: 1 - (embedding <=> $1)')
  .order('embedding <=> $1')
  .limit(topK)
  .params([embedding]); // need to handle parameterized queries; may require raw SQL.
```

Using `rpc` is cleaner.

### 3.4 Understanding Similarity Scores

- **Cosine similarity** (used in Pinecone when metric='cosine' and in pgvector with `<=>` after converting distance to similarity via `1 - distance`) ranges from -1 to 1. Higher is more similar.
- **Euclidean distance** – lower is more similar.
- Pinecone's `score` is already a similarity (if metric is cosine or dot product). For cosine, it's between -1 and 1.

When you see a score of 0.85, it means strong similarity.

### 3.5 Advanced Querying

#### Pagination
Pinecone doesn't support pagination directly (you can fetch more than needed and slice). For pgvector, you can use `offset` with `limit` but it may be inefficient without proper ordering.

#### Threshold filtering
Filter results below a certain similarity:
```javascript
const matches = results.matches.filter(m => m.score > 0.7);
```

#### Hybrid search (vector + keyword)
In pgvector, you can combine with full‑text search as shown in the previous guide. Pinecone offers hybrid via sparse vectors or the integrated embedding index that supports keyword boosting.

## 4. Indexing for Performance

Vector search speed depends on indexing. Without an index, the database performs a brute‑force scan (slow for large datasets).

### 4.1 Pinecone Indexing
Pinecone automatically builds and maintains indexes (using HNSW algorithm). You don't need to do anything – just create an index with the desired dimension and metric. However, you can configure:
- **Pods**: more pods = higher throughput.
- **Replicas**: for high availability.

### 4.2 pgvector Indexing
You must manually create an index for performance on large tables. Two index types:

#### IVFFlat (Inverted File with Flat vectors)
- Approximate index, faster build, good for most use cases.
- Requires a training step (but you can build without training first).

```sql
-- Create index using IVFFlat with cosine distance
create index on document_chunks 
using ivfflat (embedding vector_cosine_ops)
with (lists = 100); -- lists = sqrt(rows) for best performance
```

#### HNSW (Hierarchical Navigable Small World)
- Faster query, slower build, more accurate.
- Available in pgvector 0.5.0+.

```sql
create index on document_chunks 
using hnsw (embedding vector_cosine_ops);
```

**Important**: Always create an index after inserting data to avoid rebuilding. For large datasets, build index with increased maintenance_work_mem.

## 5. Best Practices

- **Batch inserts**: 100–500 vectors per request.
- **Use concurrency carefully**: Limit parallel requests to avoid overwhelming the DB or hitting rate limits.
- **Cache embeddings** for frequently used queries.
- **Monitor costs**: Pinecone charges per index hour and operations; Supabase charges for database size and compute.
- **Handle errors gracefully**: Implement retries for transient failures.
- **Keep metadata small**: Large metadata increases storage and transfer.

## 6. Complete End‑to‑End Example

Let's walk through a simple RAG‑ready flow:

```javascript
import { OpenAI } from 'openai';
import { Pinecone } from '@pinecone-database/pinecone';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index('my-docs');

// Step 1: Store a document chunk
async function storeChunk(id, text, metadata) {
  const embedding = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: text
  });
  
  await index.upsert([{
    id,
    values: embedding.data[0].embedding,
    metadata: { ...metadata, text } // include text for retrieval
  }]);
}

// Step 2: Query similar chunks
async function searchChunks(query) {
  const embedding = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: query
  });
  
  const results = await index.query({
    vector: embedding.data[0].embedding,
    topK: 5,
    includeMetadata: true
  });
  
  return results.matches.map(m => ({
    text: m.metadata.text,
    score: m.score,
    source: m.metadata.source
  }));
}

// Step 3: Generate answer (simplified)
async function ask(question) {
  const chunks = await searchChunks(question);
  const context = chunks.map(c => c.text).join('\n\n');
  
  const completion = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [
      { role: 'system', content: 'Answer based on context.' },
      { role: 'user', content: `Context:\n${context}\n\nQuestion: ${question}` }
    ]
  });
  
  return completion.choices[0].message.content;
}
```

## 7. Troubleshooting Common Issues

| Issue | Likely Cause | Solution |
|-------|--------------|----------|
| Query returns no results | Vectors not inserted or wrong dimension | Check index stats; verify embedding model matches dimension |
| Very slow queries on pgvector | Missing index | Create IVFFlat or HNSW index |
| High latency in Pinecone | Index pod size too small | Scale up pods or add replicas |
| Metadata filters not working | Filter syntax incorrect | Check Pinecone filter docs; in pgvector, ensure JSONB indexing |
| Similarity scores are negative | Using distance instead of similarity | Convert: `similarity = 1 - distance` for cosine |
