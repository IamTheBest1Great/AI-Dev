# 📘 Multi-Query Retrieval — Complete Guide

Multi-query retrieval is an advanced RAG technique that addresses a fundamental limitation of single‑query semantic search: **a user's question may have multiple interpretations, or the relevant information may be scattered across different contexts.** By generating multiple queries from the original question and retrieving for each, you dramatically improve recall and ensure you capture all relevant pieces.

This guide explains what multi-query retrieval is, why it works, and how to implement it in your RAG pipeline.

---

## 1. What Is Multi‑Query Retrieval?

Instead of taking the user's question as a single query, you:

1. **Expand** the user's question into several similar but distinct questions using an LLM.
2. **Retrieve** documents for each generated query.
3. **Combine** all retrieved results, often removing duplicates and re‑ranking.

The idea is that a single embedding may not perfectly capture the full semantic range of the user's intent. By generating variants, you cover different angles and phrasing, leading to higher recall.

**Example:**

- **Original query:** "How does photosynthesis work in plants?"
- **Generated queries:**
  - "Explain the process of photosynthesis"
  - "What are the steps of photosynthesis in plants?"
  - "How do plants convert sunlight into energy?"
  - "Describe the light and dark reactions of photosynthesis"

Even if the original query's embedding misses some relevant documents, one of the variants might hit the mark.

---

## 2. Why Multi‑Query Retrieval?

- **Improved recall** – Captures documents that are semantically similar but phrased differently.
- **Coverage of sub‑topics** – User questions often bundle multiple aspects; multi‑query disentangles them.
- **Mitigates poor embedding** – If the embedding model performs poorly on certain phrasing, variants can compensate.
- **Robustness** – Reduces the impact of poorly worded or ambiguous questions.

---

## 3. How It Works: Step‑by‑Step

```
┌─────────────┐
│ User Query  │
└──────┬──────┘
       ▼
┌─────────────┐
│ LLM Expands │   (prompt: generate 3‑5 variants)
│   Query     │
└──────┬──────┘
       │
       ├── Query 1 ──► Retrieve ──┐
       ├── Query 2 ──► Retrieve ──┤
       ├── Query 3 ──► Retrieve ──┘
       └── ...      ──►
       ▼
┌─────────────┐
│   Combine   │   (unique union, RRF, etc.)
│   Results   │
└──────┬──────┘
       ▼
┌─────────────┐

└──────┬──────┘
       ▼
┌─────────────┐
│  Generate   │
│   Answer    │
└─────────────┘
```

## 4. Generating Multiple Queries

You need an LLM to produce query variations. Provide a clear prompt with examples.

### 4.1 Prompt Design

```javascript
const generateQueriesPrompt = `
You are an AI assistant tasked with generating multiple search queries to help retrieve relevant information.

Original question: {question}

Generate 5 different versions of this question that capture different aspects or phrasing. 
Make them diverse but still focused on the same topic.
Return the queries as a JSON array of strings.
`;
```

**Example output:**
```json
[
  "What is the process of photosynthesis?",
  "Explain how photosynthesis works in plants",
  "Describe the light and dark reactions of photosynthesis",
  "How do plants convert sunlight into chemical energy?",
  "What are the steps involved in photosynthesis?"
]
```

### 4.2 Implementation with OpenAI

```javascript
import OpenAI from 'openai';

const openai = new OpenAI();

async function generateMultiQueries(question, numQueries = 5) {
  const prompt = `
Generate ${numQueries} different versions of the following question to help with document retrieval. 
Make them diverse in wording and perspective, but keep the core intent.
Return as a JSON array of strings.

Question: ${question}
`;

  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [{ role: 'user', content: prompt }],
    response_format: { type: 'json_object' },
    temperature: 0.7,
  });

  const content = response.choices[0].message.content;
  const parsed = JSON.parse(content);
  // The model might return an object like { queries: [...] } or an array directly
  return Array.isArray(parsed) ? parsed : parsed.queries || [];
}
```

### 4.3 Important Considerations

- **Number of queries**: 3–5 is typical. More may increase recall but also noise and latency.
- **Diversity**: Encourage the model to vary phrasing, not just reorder words.
- **Temperature**: Use a moderate temperature (0.5–0.7) to get variety without going off‑topic.
- **Language**: If your users speak multiple languages, you can also generate queries in other languages.

## 5. Retrieving for Each Query

After generating the list of queries, you run retrieval for each one. You can do this sequentially or in parallel.

### 5.1 Parallel Retrieval

```javascript
async function retrieveForAllQueries(queries, vectorStore, topK = 5) {
  const retrievalPromises = queries.map(q => 
    vectorStore.similaritySearch(q, topK)
  );
  const results = await Promise.all(retrievalPromises);
  // results is an array of arrays (documents per query)
  return results.flat();
}
```

**Note:** Be mindful of rate limits on your vector database; you may need to limit concurrency.

### 5.2 Using LangChain's MultiQueryRetriever

LangChain provides a built‑in `MultiQueryRetriever` that does exactly this.

```javascript
import { MultiQueryRetriever } from 'langchain/retrievers/multi_query';
import { OpenAI } from '@langchain/openai';
import { PineconeStore } from '@langchain/pinecone';

const vectorStore = await PineconeStore.fromExistingIndex(
  new OpenAIEmbeddings(),
  { pineconeIndex: index }
);

const llm = new OpenAI({ modelName: 'gpt-4o-mini', temperature: 0.7 });

const retriever = MultiQueryRetriever.fromLLM({
  llm,
  retriever: vectorStore.asRetriever(5), // base retriever with topK=5
  queryCount: 5, // number of queries to generate

const docs = await retriever.getRelevantDocuments('How does photosynthesis work?');
console.log(docs); // combined unique documents
```

The `MultiQueryRetriever` handles:
- Generating queries using the LLM.
- Retrieving for each.
- Combining results (by default, it returns unique documents).

You can customize the prompt by passing a `prompt` parameter.

## 6. Combining Retrieved Results

Simply concatenating results from all queries may introduce duplicates and irrelevant documents. You need a strategy to merge and rank.

### 6.1 Unique Union

The simplest: deduplicate by document ID or content hash.

```javascript
function uniqueUnion(results) {
  const seen = new Set();
  return results.filter(doc => {
    const id = doc.id || doc.pageContent; // adjust as needed
    if (seen.has(id)) return false;
    seen.add(id);
    return true;
  });
}
```

### 6.2 Reciprocal Rank Fusion (RRF)

RRF combines rankings from multiple result sets without needing scores. For each document, its score is the sum over each query's rank: `score = Σ 1 / (k + rankᵢ)` where `k` is a constant (usually 60).

```javascript
function reciprocalRankFusion(queryResults, k = 60) {
  // queryResults: array of arrays of documents (each array sorted by relevance)
  const scoreMap = new Map(); // docId -> total score
  
  queryResults.forEach((docsForQuery) => {
    docsForQuery.forEach((doc, rank) => {
      const docId = doc.id || doc.pageContent;
      const current = scoreMap.get(docId) || 0;
      scoreMap.set(docId, current + 1 / (k + rank));
    });
  });
  
  // Sort documents by score descending
  const sorted = [...scoreMap.entries()]
    .sort((a, b) => b[1] - a[1])
    .map(([docId]) => {
      // find the original doc object (you may need to store mapping)
      return docMap.get(docId);
    });
  
  return sorted;
}
```

### 6.3 Re‑ranking After Fusion

After combining, you can optionally apply a cross‑encoder re‑ranker to the top‑N fused results to improve final ordering (see the re‑ranking guide).

## 7. Full Implementation Example

Here's a complete, self‑contained example using OpenAI, Pinecone, and a simple combination strategy.

```javascript
import OpenAI from 'openai';
import { Pinecone } from '@pinecone-database/pinecone';
import { OpenAIEmbeddings } from '@langchain/openai';
import { PineconeStore } from '@langchain/pinecone';

const openai = new OpenAI();
const pc = new Pinecone({ apiKey: process.env.PINECONE_API_KEY });
const index = pc.index('my-index');

async function generateQueries(question) {
  const prompt = `
Generate 5 different search queries to find information relevant to this question:
"${question}"
Make them varied in wording and perspective. Return as a JSON array of strings.
`;
  const response = await openai.chat.completions.create({
    model: 'gpt-4o-mini',
    messages: [{ role: 'user', content: prompt }],
    response_format: { type: 'json_object' },
    temperature: 0.7,
  });
  const parsed = JSON.parse(response.choices[0].message.content);
  return Array.isArray(parsed) ? parsed : parsed.queries;
}

async function retrieveAll(queries, vectorStore, topK = 5) {
  const results = await Promise.all(
    queries.map(q => vectorStore.similaritySearch(q, topK))
  );
  return results;
}

async function multiQueryRetrieval(question) {
  // 1. Generate queries
  const queries = await generateQueries(question);
  console.log('Generated queries:', queries);

  // 2. Setup vector store
  const embeddings = new OpenAIEmbeddings();
  const vectorStore = await PineconeStore.fromExistingIndex(embeddings, { pineconeIndex: index });

  // 3. Retrieve for each query
  const resultsPerQuery = await retrieveAll(queries, vectorStore, 5);

  // 4. Combine (simple unique union)
  const seen = new Set();
  const combined = [];
  for (const docs of resultsPerQuery) {
    for (const doc of docs) {
      const key = doc.pageContent; // or doc.id
      if (!seen.has(key)) {
        seen.add(key);
        combined.push(doc);
      }
    }
  }

  // 5. (Optional) re‑rank combined results
  return combined.slice(0, 10); // top 10
}

// Usage
const docs = await multiQueryRetrieval('How does photosynthesis work?');
console.log(`Retrieved ${docs.length} unique documents.`);
```

## 8. Best Practices and Considerations

- **Number of queries**: Start with 3–5. Too many can introduce noise and increase latency.
- **Deduplication**: Always remove duplicates; they waste context window.
- **Score preservation**: If your retriever returns scores, you can use them for weighted fusion (e.g., average score across queries).
- **Latency**: Parallel retrieval reduces total time, but be aware of rate limits.
- **Cost**: Generating multiple queries uses LLM calls and more retrieval operations. Cache frequent questions.
- **Evaluation**: Test whether multi‑query improves your specific use case. Not every dataset benefits equally.
- **Prompt engineering**: Explicitly ask for diverse phrasing to avoid near‑identical queries.

---

## 9. When to Use Multi‑Query Retrieval

- **Complex, multi‑faceted questions** – e.g., "What are the causes and treatments for diabetes?" could generate queries about causes, treatments, symptoms separately.
- **Ambiguous questions** – e.g., "How to bake a cake?" could generate "ingredients for cake", "baking steps", "cake recipes".
- **High‑recall requirements** – Legal, medical, or research domains where missing a relevant document is costly.
- **Poor embedding model performance** – If your embeddings are not great, multi‑query can compensate.

Avoid if:
- Your questions are always simple and direct.
- Latency is critical and you cannot afford the extra time.
- Your retrieval is already very high‑recall.

---

## 10. Advanced: Self‑Querying + Multi‑Query

You can combine multi‑query with a self‑querying retriever that extracts filters (e.g., date, category) from the question. For each generated query, you could also let the LLM suggest metadata filters.

LangChain's `SelfQueryRetriever` can be extended to work with multiple queries, though it's more complex. Usually, you'd apply the self‑query logic to the original question and then use the same filter for all generated queries, or let each query generate its own filter.
