# 📘 Embeddings & Vector Search — Union of All Concepts

This guide merges the foundational theory of word embeddings with practical implementation details for vector search, chunking, and similarity metrics that power modern AI applications like RAG, semantic search, and recommendation systems.

---

## 1. What Are Embeddings?

### Text → Dense Vector Transformation
An **embedding** is a **dense vector** (a list of real numbers) that represents the semantic meaning of a piece of text. It is a numerical “coordinate” that places each piece of text in a high‑dimensional **meaning space**. Words or sentences with similar meanings end up close together.

**Example of word vectors**:
```
"cat" → [0.23, -0.56, 0.12, ..., 0.07]   (e.g., 300 numbers)
"dog" → [0.21, -0.52, 0.14, ..., 0.09]   (similar numbers)
"car" → [0.87, 0.21, -0.45, ..., -0.03]  (different numbers)
```

Similarly:
```
"king"  → [0.23, -1.02, 0.45, ..., 0.11]
"queen" → [0.21, -0.98, 0.47, ..., 0.09]
"apple" → [0.65, 0.12, -0.33, ..., 0.72]
```

Unlike sparse representations (one‑hot encoding), dense vectors pack meaning into a relatively low‑dimensional space.

---

## 2. Why Do We Need Embeddings?

Computers only understand numbers. Traditional representations like **one‑hot encoding** treat every word as independent, with every pair of words equally different:

```
"cat" → [1, 0, 0, 0, …]
"dog" → [0, 1, 0, 0, …]
```

In one‑hot, the distance between *cat* and *dog* is the same as between *cat* and *car* — the model cannot learn that *cat* and *dog* are both pets. Embeddings solve this by placing words in a continuous space where **distance directly reflects semantic similarity**.

---

## 3. How Are Embeddings Created?

Embeddings are learned by training neural networks on massive amounts of text. The model learns to predict words from context (or vice versa).

### Key algorithms
- **Word2Vec** (2013) – uses “skip‑gram” (predict surrounding words from a target) or “CBOW” (predict target from surrounding words).
- **GloVe** (2014) – factorizes word‑co‑occurrence matrices.
- **FastText** – handles out‑of‑vocabulary words by using subword information.
- **Modern transformers** (BERT, GPT, sentence‑transformers) – produce **contextual embeddings**: the same word gets different vectors depending on its context (e.g., “bank” in “river bank” vs. “money bank”).

**Intuition**: A model sees millions of sentences like
> “I drank a glass of ___.”
It learns that *water*, *milk*, *juice* often fill the blank, so these words receive similar vectors.

### Dimensionality
Typical sizes: 50, 100, 300, 384, 768, 1024, 1536 (OpenAI’s `text-embedding-3-small`), 3072 (OpenAI’s `text-embedding-3-large`). Higher dimensions capture more nuance but require more storage and computation.

---

## 4. Types of Embeddings

| Type | Description | Example Models |
|------|-------------|----------------|
| **Static** | Each word has one fixed vector, regardless of context. | Word2Vec, GloVe, FastText |
| **Contextual** | The same word gets different vectors based on surrounding text. | BERT, GPT, OpenAI embeddings, Cohere embed |

For most modern applications (RAG, semantic search), **contextual embeddings** are standard because they handle polysemy and capture richer meaning.

## 5. Fundamental Properties of Embeddings

### Similarity → Proximity
The core property: **distance (or similarity) in vector space equals semantic similarity**. Similar concepts → high cosine similarity / small Euclidean distance. This lets you:
- Find documents relevant to a query by comparing vectors.
- Cluster documents by topic.
- Recommend similar items.

### Distance Metrics (overview)
- **Cosine similarity** – measures the angle between vectors (most common). Range: -1 to 1.
- **Euclidean distance (L2)** – straight‑line distance. Sensitive to magnitude.
- **Dot product** – often used when vectors are normalised to unit length.

---

## 6. Similarity Metrics in Depth

When you have two embedding vectors, you must choose a metric. Below are the three primary choices with implementations and guidance.

### Cosine Similarity (Most Common)
Measures the **angle** between vectors; focuses on direction (meaning) not magnitude.

**Formula**:  
$$\text{cos\_sim}(A,B) = \frac{A \cdot B}{\|A\| \times \|B\|}$$

Range: -1 (opposite meaning) to 1 (identical meaning). For normalised text embeddings, typically 0 to 1.

**Implementation (JavaScript)**:
```javascript
function cosineSimilarity(vecA, vecB) {
  const dot = vecA.reduce((sum, a, i) => sum + a * vecB[i], 0);
  const normA = Math.sqrt(vecA.reduce((sum, a) => sum + a * a, 0));
  const normB = Math.sqrt(vecB.reduce((sum, b) => sum + b * b, 0));
  return dot / (normA * normB);
}
```

### Dot Product (for Normalised Vectors)
If vectors have unit length (`||A|| = 1`, `||B|| = 1`), cosine similarity simplifies to the dot product.

`dot(A,B) = Σ A_i × B_i`

Range for unit vectors: -1 to 1. Many vector databases (Pinecone, etc.) use dot product because it’s faster.

```javascript
function dotProduct(vecA, vecB) {
  return vecA.reduce((sum, a, i) => sum + a * vecB[i], 0);
}
```

### Euclidean Distance (L2)
Measures the straight‑line distance between vectors. Sensitive to magnitude.

**Formula**:  
$$\text{euclidean}(A,B) = \sqrt{ \sum (A_i - B_i)^2 }$$

Range: 0 (identical) to ∞ (very dissimilar).

```javascript
function euclideanDistance(vecA, vecB) {
  const sumSq = vecA.reduce((sum, a, i) => {
    const diff = a - vecB[i];
    return sum + diff * diff;
  }, 0);
  return Math.sqrt(sumSq);
}
```

### Metric Comparison Table

| Metric | Range | Sensitivity | Use Case |
|--------|-------|-------------|----------|
| Cosine similarity | -1 to 1 | Direction only | Semantic search (default) |
| Dot product (normalised) | -1 to 1 | Direction only | Optimised vector DBs |
| Euclidean distance | 0 to ∞ | Direction + magnitude | When magnitude matters (e.g., anomaly detection) |

**Practical tip**: Most vector databases let you choose the metric at index creation. For text, use **cosine** or **dot product** (if vectors are normalised). Stick with cosine for clarity.

## 7. Vector Arithmetic – Capturing Relationships

Embeddings capture relational structure. The famous example:

```
vector("king") - vector("man") + vector("woman") ≈ vector("queen")
```

**How it works**:
1. `king` captures royalty + male.
2. Subtract `man` → remove male connotations.
3. Add `woman` → add female connotations.
4. Result is very close to `queen`.

Other examples:
- `paris - france + italy ≈ rome`
- `pasta - italy + japan ≈ sushi`

This means you can perform analogies and manipulate conceptual dimensions directly in vector space.

## 8. Generating Embeddings – Practical Implementation

### OpenAI `text-embedding-3-small` (1536 dimensions)
Efficient and suitable for most RAG and semantic search tasks.

**Basic usage (JavaScript)**:
```javascript
import OpenAI from 'openai';

const openai = new OpenAI({ apiKey: process.env.OPENAI_API_KEY });

async function embed(text) {
  const response = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: text,
  });
  return response.data[0].embedding; // length 1536
}
```

**Cost** (as of 2026): ~$0.02 per million tokens (input). For a typical 1000‑token document, ~$0.00002 per embedding.

**Other OpenAI models**:
- `text-embedding-3-large` – 3072 dimensions, higher accuracy, ~$0.13 per million tokens.
- `text-embedding-ada-002` – legacy 1536‑dim model (being phased out).

### Cohere Embed Models (1024 dimensions)
Offers both English and multilingual models, with optimised input types.

```javascript
import { CohereClient } from 'cohere-ai';

const cohere = new CohereClient({ token: process.env.COHERE_API_KEY });

async function embedCohere(text) {
  const response = await cohere.embed({
    texts: [text],
    model: 'embed-english-v3.0',
    inputType: 'search_document',  // or 'search_query' for user queries
  });
  return response.embeddings[0];
}
```

**Input types**: Always use `search_document` for stored chunks and `search_query` for the user’s question to optimise retrieval.

### Hugging Face / Sentence‑Transformers (Python example)
Lightweight, local models. `all-MiniLM-L6-v2` gives 384‑dimensional vectors.

```python
from sentence_transformers import SentenceTransformer

model = SentenceTransformer('all-MiniLM-L6-v2')
sentences = ["This is an example sentence", "Each sentence is converted"]
embeddings = model.encode(sentences)
print(embeddings.shape)  # (2, 384)
```

### Batch Embedding for Efficiency
Instead of embedding texts one by one, send multiple texts in a single API call.

**Why batch?**
- Lower cost (fewer API calls).
- Lower latency (amortised round‑trip time).
- Higher throughput (providers often give higher rate limits for batches).

**OpenAI batch** (up to 2048 texts per call):
```javascript
async function embedBatch(texts) {
  const response = await openai.embeddings.create({
    model: 'text-embedding-3-small',
    input: texts,  // array of strings
  });
  return response.data.map(item => item.embedding);
}

// Process 1000 chunks
const chunkTexts = getChunks(longDocument);
const embeddings = await embedBatch(chunkTexts);
```

**Cohere batch** (up to 96 texts per call):
```javascript
const response = await cohere.embed({
  texts: chunkTexts,
  model: 'embed-english-v3.0',
  inputType: 'search_document',
});
```

**Practical batch sizes**: OpenAI: 100–500 per batch (max 2048). Cohere: 50–96. Always check documentation for exact limits. Implement exponential backoff and retry for rate limit handling.

## 9. Chunking Strategies for Long Texts

Before embedding a long document, you must split it into smaller **chunks**. The strategy directly affects retrieval accuracy.

### Fixed‑Size with Overlap
Split by a fixed number of characters/tokens, with some overlap between consecutive chunks.

```javascript
function fixedSizeChunk(text, chunkSize = 512, overlap = 64) {
  const chunks = [];
  let start = 0;
  while (start < text.length) {
    const end = Math.min(start + chunkSize, text.length);
    chunks.push(text.slice(start, end));
    start += chunkSize - overlap;
  }
  return chunks;
}
```
**Pros**: Simple, fast.  
**Cons**: May cut sentences or paragraphs in half.

### Recursive (Hierarchical Separators)
Uses a hierarchy of separators (`\n\n`, `\n`, `.`, ` `) to keep semantic units together.

```javascript
import { RecursiveCharacterTextSplitter } from 'langchain/text_splitter';

const splitter = new RecursiveCharacterTextSplitter({
  chunkSize: 512,
  chunkOverlap: 64,
  separators: ['\n\n', '\n', '.', ' ', ''],  // from most to least semantic
});
const chunks = await splitter.splitText(text);
```
**Pros**: Respects document structure; more coherent chunks.  
**Cons**: Slightly slower; may still split at non‑ideal points if no separator fits.

### Semantic (by Meaning Shift)
Uses embeddings to detect where topics change; splits when similarity between consecutive sentences drops below a threshold.

```javascript
async function semanticChunking(sentences, threshold = 0.7) {
  const chunks = [];
  let currentChunk = [sentences[0]];
  let prevEmbedding = await embed(sentences[0]);
  
  for (let i = 1; i < sentences.length; i++) {
    const currEmbedding = await embed(sentences[i]);
    const sim = cosineSimilarity(prevEmbedding, currEmbedding);
    if (sim > threshold) {
      currentChunk.push(sentences[i]);
    } else {
      chunks.push(currentChunk.join(' '));
      currentChunk = [sentences[i]];
    }
    prevEmbedding = currEmbedding;
  }
  if (currentChunk.length) chunks.push(currentChunk.join(' '));
  return chunks;
}
```
**Pros**: Topically coherent chunks (one “idea” per chunk).  
**Cons**: Expensive (requires many embedding calls); use for high‑quality needs or offline processing.

### Token‑Based (Most Accurate)
Uses a tokenizer (e.g., `tiktoken`) to ensure exact token counts, never exceeding the model’s limit.

```javascript
import { encoding_for_model } from 'tiktoken';

function tokenBasedChunk(text, maxTokens = 512, overlap = 64) {
  const enc = encoding_for_model('text-embedding-3-small');
  const tokens = enc.encode(text);
  const chunks = [];
  let start = 0;
  
  while (start < tokens.length) {
    const end = Math.min(start + maxTokens, tokens.length);
    const chunkTokens = tokens.slice(start, end);
    chunks.push(enc.decode(chunkTokens));
    start += maxTokens - overlap;
  }
  enc.free();
  return chunks;
}
```
**Pros**: Exact control; avoids token limit errors and wasted capacity.  
**Cons**: Requires tokenizer library; slightly slower.

### Overlap Importance (10–20%)
Overlap ensures that critical information cut at a boundary appears in the next chunk. Without it, a sentence split across chunks can lose context (“The quick brown fox jumps over the” / “lazy dog.”). Recommendation: 10–20% of chunk size (e.g., 50–100 tokens for a 512‑token chunk).

## 10. Chunk Quality and Best Practices

Even with a good strategy, the actual quality of chunks determines retrieval success.

### Optimal Chunk Size (256–512 tokens)

| Size | Effect |
|------|--------|
| < 128 tokens | Very specific, may miss broader context. High precision, low recall. |
| **256–512 tokens** | **Sweet spot** for most RAG apps – balanced precision/recall. |
| 512–1024 tokens | More context, but may include irrelevant information. |
| > 1024 tokens | Too noisy; relevant information diluted. Risk of exceeding context windows. |

### Context Continuity
A chunk should be a **coherent unit of thought**:
- **Split** at paragraph breaks, section boundaries, or clear topic shifts.
- **Never split** mid‑sentence, inside a bullet point, or within a code block.

**Red flags**: chunk starts with a pronoun whose antecedent is in the previous chunk; a list broken across chunks; code block split in two.

### Boundary Detection – Where to Split
Order cut points from most to least semantic:

```javascript
separators: ['\n\n', '\n', '.', '?', '!', ' ', '']
```

- **Markdown headings** (`##`, `###`) – excellent split points.
- **Paragraph breaks** (`\n\n`) – safe splits.
- **Sentence boundaries** (`.`, `!`, `?` followed by space and capital) – acceptable.
- **List markers** – keep entire list in one chunk when possible.
- **Code blocks** – never split inside.

### Chunk Quality Checklist

| Criteria | How to Evaluate |
|----------|-----------------|
| **Length** | 80% of chunks within target range (e.g., 200–600 tokens) |
| **Boundary cleanliness** | No split mid‑sentence in >95% of chunks |
| **Topic coherence** | Each chunk covers a single idea/topic |
| **Redundancy** | Overlap not excessive (>20%) |
| **Completeness** | No pronoun without antecedent within chunk |

### Evaluation Tools
- **LangChain** – `CharacterTextSplitter`, `RecursiveCharacterTextSplitter`, `TokenTextSplitter` with metrics.
- **LlamaIndex** – `SentenceSplitter`, `SemanticSplitterNodeParser`.
- Manual check: print random chunks and rate them 1–5.

## 11. Use Cases in AI Applications

- **Semantic Search** – search by meaning, not keywords.  
  Query “affordable lodging in Paris” retrieves documents about “cheap hotels near Eiffel Tower” even without exact word matches.
- **Retrieval‑Augmented Generation (RAG)** – convert user question to embedding, find relevant chunks from a vector DB, feed to an LLM.
- **Clustering** – group similar documents (e.g., customer feedback by topic).
- **Classification** – use embeddings as feature vectors for classifiers.
- **Recommendation** – recommend items with embeddings similar to a user’s past interactions.

## 14. Key Takeaways

- **Embeddings = dense vectors** that encode semantic meaning.
- **Similar meaning → similar vectors** (high cosine similarity).
- They are generated by neural networks trained on large text corpora (Word2Vec, GloVe, FastText, Transformers).
- **Contextual embeddings** (BERT, GPT, OpenAI, Cohere) are now standard.
- **Similarity metrics** – cosine is the default for text; dot product for normalised vectors; Euclidean when magnitude matters.
- **Vector arithmetic** allows relational reasoning (`king − man + woman ≈ queen`).
- **Chunking** is essential for long documents; use 256–512 tokens with 10–20% overlap and respect semantic boundaries.
- **Chunk quality** directly impacts retrieval – check coherence, completeness, and boundary cleanliness.
- Embeddings are the foundation of **vector databases**, **RAG**, and **semantic search**.
